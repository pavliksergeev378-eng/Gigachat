# Агент «Меры информационной безопасности» — технический гайд

> **Для кого этот документ:** администратор/разработчик платформы GigaChat, который
> разворачивает, наполняет и сопровождает агента. Если вы **обычный пользователь**
> (специалист ИБ, который просто формирует меры по нарушению) — вам нужен
> [**гайд пользователя**](Compliance-User-Guide.md), он проще и без техники.

Агент принимает выявленное **нарушение информационной безопасности** и формирует
**компенсирующие меры** (организационные, технические, административные,
контролирующие) с обоснованием, ссылками на нормативные требования, оценкой
**остаточного риска** и рекомендуемыми **сроками**. Меры берутся из выверенной
экспертами **базы знаний** в PostgreSQL — поэтому ответы воспроизводимы, со
ссылками на источник и **без «галлюцинаций» модели**.

> **Где в дашборде:** раздел «Алгоритмы» → карточка **«Меры информационной безопасности»**.

## 1. Зачем агент нужен

При проверках ИБ (аудит, заключение комиссии, акт) выявляются нарушения. По
каждому нужно подобрать **компенсирующие меры**: что конкретно сделать, чтобы
закрыть риск, с обоснованием и ссылкой на нормативное требование. Вручную это
долго и неединообразно — разные специалисты предлагают разные меры.

Агент делает это **единообразно, быстро и прослеживаемо**:
- по тексту нарушения находит **типовое нарушение** в базе знаний;
- подтягивает закреплённые за ним **меры**, **нарушенные требования**, **риски**;
- собирает отчёт: категория, уровень риска, меры по 4 типам (наименование +
  описание + ожидаемый эффект + обоснование), **остаточный риск**, **сроки**;
- сохраняет инцидент в историю; утверждённое решение пополняет базу как
  обучающий кейс.

**Главный принцип:** меры берутся из экспертной базы, а не выдумываются моделью.
Это даёт прослеживаемость (у каждого требования — источник) и повторяемость.

## 2. Как это работает — за 1 минуту

```
Пользователь вводит нарушение
        │
        ▼
[Матч с базой знаний]  ──найдено──►  меры + требования + риски этого нарушения
        │                                        │
     не найдено                                  ▼
        │                            [Сборка отчёта: ≥3 меры, остаточный риск, сроки]
        ▼                                        │
[Базовые меры ≥3 + пометка «уточните»] ──────────┘
        │
        ▼
[Сохранение инцидента (status=draft)]  ──►  ответ пользователю
        │
   «Утвердить» (опц.)
        ▼
[Инцидент → approved] + [кейс в базу знаний]
```

Всё **детерминированно** (только PostgreSQL + код-ноды), без эмбеддера/LLM —
это сознательный выбор для задачи, где важны точность источников и отсутствие
выдумок (расширение LLM-слоем — [§12](#12-расширение-семантика-и-llm-офисный-слой)).

## 5. Алгоритм подбора мер (детально)

Ветка `analyze` (узлы по порядку):

1. **Загрузить нарушения** — `SELECT id, title, category, processes, description, criticality FROM kb_violations`
   (таблица небольшая — берём целиком).
2. **Сопоставить** (код-нода) — текстовый матч без эмбеддера:
   - входящее нарушение + описание + результаты проверок токенизируются
     (значимые слова ≥ 4 символов);
   - для каждого типового нарушения считается пересечение токенов с
     `title + category + description`;
   - **оценка = `пересечение / √(размер запроса)`** — нормировка по запросу,
     чтобы подробно описанные нарушения в базе не штрафовались (раньше нормировали
     по длине записи — и хорошо документированные записи не находились);
   - берётся лучший матч; **порог = 1.0**. Ниже порога — «не найдено».
3. **Загрузить связи** — одним запросом (`json_agg`) подтягиваются **меры**,
   **требования** (с источником `norm_ref`) и **риски** выбранного нарушения.
4. **Сборка результата** (код-нода):
   - меры группируются по 4 типам, каждой добавляется **обоснование**;
   - **гарантия ≥ 3 мер**: если связано меньше — добиваются базовыми
     (назначить ответственного → включить в план проверок → разработать регламент);
   - **остаточный риск** = на уровень ниже входного (critical→high→medium→low);
   - **сроки** считаются по уровню риска (critical: 7/14/30 дн, high: 14/30, medium: 30, low: 45);
   - формируется отчёт строго в формате, который ждёт фронтенд.
5. **Сохранить инцидент** — запись в `compliance_incidents` со `status='draft'`
   (включая `check_results` — выдержки из акта).

**Если типовое нарушение не найдено** — агент всё равно возвращает **базовые
меры (≥3)** и честно помечает в поле `note`: «не найдено в базе знаний,
рекомендуется уточнить через эксперта / пополнить базу». Так агент всегда что-то
предлагает, но не выдаёт догадку за точный матч.

Ветка `approve` — пишет утверждённый набор мер в `kb_cases` **и** переводит
исходный инцидент в `status='approved'` (см. [§7](#7-жизненный-цикл-инцидента-draft--approved)).
Запрос `ping` (для health-check дашборда) — отвечает `pong`.

## 6. База знаний: модель и наполнение

Качество ответов = качество базы. Главная ценность — **связи** между нарушениями,
мерами, требованиями и рисками (их заполняют эксперты ИБ).

### 6.1. Таблицы

| Таблица | Назначение |
|---|---|
| `kb_violations` | типовые нарушения (`title`, `category`, `processes`, `description`, `criticality`) |
| `kb_measures` | меры (`type` ∈ organizational/technical/administrative/controlling, `name`, `description`, `expected_effect`) |
| `kb_requirements` | нормативные требования (`text` + `norm_ref` — источник) |
| `kb_risks` | типовые риски |
| `kb_violation_measure` | связь нарушение ↔ меры |
| `kb_violation_requirement` | связь нарушение ↔ требования |
| `kb_violation_risk` | связь нарушение ↔ риски |
| `kb_norms` | корпус нормативки для семантического поиска (наполняется загрузчиком; нужен офисному LLM-слою, [§12](#12-расширение-семантика-и-llm-офисный-слой)) |
| `kb_cases` | утверждённые решения (самообогащение базы; будущий few-shot) |
| `compliance_incidents` | история обработанных нарушений (вход + отчёт + статус) |

### 6.2. Как добавить новое типовое нарушение с мерами

```sql
-- 1) нарушение
INSERT INTO kb_violations (title, category, processes, description, criticality) VALUES
('Открыт доступ к БД из внешней сети', 'Сетевая безопасность',
 'Защита периметра; Управление доступом',
 'СУБД слушает на внешнем интерфейсе без ограничения по подсети.', 'high')
ON CONFLICT (title) DO NOTHING;

-- 2) меры (если их ещё нет)
INSERT INTO kb_measures (type, name, description, expected_effect) VALUES
('technical','Ограничить доступ к СУБД по подсети','Настроить pg_hba.conf / firewall на доверенные адреса.','Закрытие внешнего доступа к данным.'),
('controlling','Периодически сканировать открытые порты','Регулярный аудит доступности портов извне.','Своевременное выявление повторного открытия.')
ON CONFLICT (name) DO NOTHING;

-- 3) связь нарушение → меры (по названиям)
INSERT INTO kb_violation_measure (violation_id, measure_id)
SELECT v.id, m.id FROM kb_violations v JOIN kb_measures m ON TRUE
WHERE v.title = 'Открыт доступ к БД из внешней сети'
  AND m.name IN ('Ограничить доступ к СУБД по подсети','Периодически сканировать открытые порты')
ON CONFLICT DO NOTHING;

-- 4) (опц.) требование и риск + связи — аналогично, через kb_requirements/kb_risks
```

> **Рекомендация:** держите **≥ 3 точечные меры** на нарушение. Если связано
> меньше, агент добьёт базовыми — но экспертные меры качественнее.
>
> **Подсказка по матчингу:** чтобы нарушение находилось, в его `title`
> (и `description`) должны быть **те же значимые слова**, которыми пользователь
> опишет проблему («журналирование», «резервное копирование», «антивирус»…).

### 6.3. Сид «из коробки»

`init-db.sql` кладёт пример: **10 мер**, **4 нарушения** (резервное копирование,
парольная политика, антивирус, журналирование), их связи и **1 кейс**. Это
демонстрационный набор — замените реальной нормативкой и связями организации.

## 7. Жизненный цикл инцидента (draft → approved)

```
analyze ──► INSERT compliance_incidents (status='draft') RETURNING id
                                   │
              ответ фронту: { …отчёт…, incident_id: <id> }
                                   │
                     пользователь жмёт «Утвердить»
                                   │
approve { result, incident_id } ──► WITH ins AS (INSERT kb_cases …)
                                    , upd AS (UPDATE compliance_incidents
                                              SET status='approved'
                                              WHERE id=incident_id AND user_id=…)
                                    SELECT … ;  → { ok: true }
```

- `incident_id` возвращается из `analyze` и передаётся фронтом в `approve` —
  это связывает «утверждение» с конкретным инцидентом.
- `UPDATE` дополнительно фильтрует по `user_id` — чужой инцидент утвердить нельзя.
- Без `incident_id` (например, утверждают пример) — кейс в `kb_cases` пишется,
  а статус инцидента не трогается (нет такого инцидента в БД — это нормально).

## 8. Контракт API (вход/выход)

**Эндпоинт:** `POST {N8N_BASE}/webhook/compliance`

**Анализ — запрос:**
```json
{
  "action": "analyze",
  "token": "<64-символьный токен сессии>",
  "violation": "Отсутствует резервное копирование критически важных серверов",
  "description": "Для критичных серверов не настроено копирование…",
  "control_object": "Серверная инфраструктура",
  "criticality": "critical",
  "risk_level": "critical",
  "check_results": "выдержки из акта…"
}
```
> Сервер обрезает длины: `violation` ≤ 500, `control_object` ≤ 300,
> `description`/`check_results` ≤ 5000 (защита от переполнения VARCHAR и тяжёлой
> токенизации). Токен принимается в любом регистре (нормализуется к нижнему).

**Анализ — ответ** (1-в-1 как ждёт фронтенд):
```json
{
  "violation": "…",
  "category": "Резервное копирование",
  "criticality": "critical",
  "processes": ["Восстановление", "Непрерывность"],
  "risk_level": "critical",
  "residual_risk": "high",
  "requirements": [{ "text": "…", "source": "…" }],
  "measures": {
    "organizational": [{ "name": "…", "description": "…", "effect": "…", "justification": "…" }],
    "technical": [ … ], "administrative": [ … ], "controlling": [ … ]
  },
  "deadlines": [{ "measure": "…", "term": "14 дней" }],
  "matched": true,
  "note": "",
  "incident_id": 42
}
```

**Утвердить:** `{ "action": "approve", "token": "…", "result": <объект результата>, "incident_id": 42 }`
→ `{ "ok": true, "response": "Кейс добавлен в базу знаний" }`.

**Ping (health-check):** `{ "message": "ping" }` → `{ "response": "pong", "auth_required": false }`.

**Без токена / неверный токен:** любой `analyze`/`approve` → `{ "response": "auth_required", "auth_required": true }`.

**Уровни:** `low` · `medium` · `high` · `critical`.
**Типы мер:** `organizational` · `technical` · `administrative` · `controlling`.

## 9. Развёртывание с нуля

### Шаг 1. База данных
```bash
psql -U postgres -c "DROP DATABASE IF EXISTS ai_agent;"
psql -U postgres -c "CREATE DATABASE ai_agent;"
psql -U postgres -d ai_agent -f "База данных/init-db.sql"
```
Проверка: `SELECT count(*) FROM kb_measures;` → `10` (сид).
`init-db.sql` идемпотентен (`CREATE IF NOT EXISTS` + `INSERT … ON CONFLICT`) —
можно безопасно накатывать поверх существующей БД (таблицы и сид не перезапишутся).

### Шаг 2. Импорт воркфлоу в n8n
```powershell
.\import-workflows.ps1 -Filter compliance
```
Импортирует **оба** воркфлоу — «Compliance-агент. Поток» (`compliance.json`) и
«Compliance KB. Загрузчик базы знаний» (`compliance-kb.json`) — и привяжет
Postgres-credential по имени `Postgres` (подробнее — `Import-Workflows-Guide.md`).

### Шаг 3. Активация
```powershell
.\activate-workflows.ps1
```
Активирует **оба** воркфлоу: вебхуки `compliance` и `compliance-kb` принимают POST;
карточка на дашборде зелёная (health-check отвечает `pong`).

### Шаг 3.1. (Опционально) Роль администратора
Пополнять Базу знаний может **любой вошедший** пользователь — вкладка «База знаний»
видна всем, отдельных действий не требуется. Колонка `auth_users.is_admin` остаётся
в схеме (на будущее), но для доступа к БЗ больше не нужна.

### Шаг 3.2. (Офис) Векторизовать существующую БЗ
Сид `kb_violations` приходит без эмбеддингов. Чтобы заработал семантический матч,
переиндексируйте их **через офисный эмбеддер** — проще всего открыть каждое
нарушение во вкладке «База знаний» и пересохранить (на сохранении считается
вектор). Либо разово прогнать тексты через `/embed` и записать `embedding`.
Без этого шага матч работает по тексту (token-overlap) — это нормальный фолбэк.

### Шаг 4. Проверка с дашборда
`GigaChat-Platform.html` → «Алгоритмы» → «Меры информационной безопасности» →
ввести нарушение → «Сформировать меры». Кнопка **«Пример»** работает всегда
(демо в браузере, без бэкенда).

## 10. Проверка работоспособности (E2E)

**Вебхук живой (ping):**
```powershell
Invoke-RestMethod -Uri 'http://localhost:5678/webhook/compliance' -Method Post `
  -Body (@{message='ping'} | ConvertTo-Json) -ContentType 'application/json'
# → { response = pong; auth_required = False }
```

**Авторизация (без токена):**
```powershell
Invoke-RestMethod -Uri 'http://localhost:5678/webhook/compliance' -Method Post `
  -Body (@{action='analyze'; violation='тест'} | ConvertTo-Json) -ContentType 'application/json'
# → { response = auth_required; auth_required = True }
```

**Полный анализ + утверждение** (нужен валидный токен из `auth_sessions` и
наполненная база) — залогиньтесь через `login.html`, затем «Сформировать меры»
по реальному нарушению, потом «Утвердить». Проверьте в БД:
```sql
SELECT id, status, left(check_results,20) FROM compliance_incidents ORDER BY id DESC LIMIT 1;  -- status='approved', check_results заполнен
SELECT count(*) FROM kb_cases;  -- вырос после «Утвердить»
```

## 11. Безопасность и изоляция

- **Авторизация.** Каждый запрос несёт `token`; воркфлоу проверяет его в
  `auth_sessions` (живая, не истёкшая сессия). Без валидного — `auth_required`,
  фронт уводит на `login.html`. При **сбое БД** во время проверки токена узел
  «Token result» возвращает `database_unavailable` (а не `auth_required`) —
  чтобы временный сбой БД не разлогинивал пользователя.
- **История по пользователю.** В `compliance_incidents` пишется `user_id`
  владельца токена. Утвердить можно только **свой** инцидент (`UPDATE … AND user_id=…`).
- **Никаких выдумок.** Меры и источники — только из базы знаний.
- **Защита ввода.** Длины обрезаются на бэкенде; на форме — `maxlength`.
- **Офлайн.** Всё работает в корпоративной сети без интернета.

## 12. ИИ-ядро: векторизация, RAG, LLM (реализовано)

Поверх детерминированного ядра работает ИИ-слой. **Всё офлайн-безопасно**: если
офисные сервисы недоступны, поток откатывается на текстовое ядро и не падает.

**Адреса сервисов** (зашиты в узлах `compliance.json`, как в RAG/document-loader):
- эмбеддер `http://130.100.94.119:8001/embed` (e5-large, 1024, без auth);
- LLM `http://130.100.95.104:8810/v1/chat/completions` (OpenAI-совместимый).

Что делает ИИ-слой в ветке analyze:
- **Гибридный матч.** Узел «Эмбеддинг нарушения» векторизует ввод; «Загрузить
  нарушения» считает близость `1 - (embedding <=> $vec)`; «Сопоставить» берёт
  лучший результат (семантический при `sim ≥ 0.80`, иначе текстовый). Эмбеддер
  недоступен → `sim=null` → чистый token-overlap (как раньше).
- **RAG по нормативке.** «RAG: нормы» — семантический поиск по `kb_norms`
  (ivfflat), top-3 фрагмента; источник (документ/раздел) добавляется в требования.
- **LLM-обогащение.** «LLM-обогащение» по нарушению + мерам из БЗ + RAG-нормам +
  few-shot из `kb_cases` формирует обоснования к мерам, при нехватке предлагает
  доп. меры (`source='ai'`, метка «предложено ИИ») и краткое резюме. **Строго на
  основе переданных данных**, источники норм не выдумывает. Сбой → `enrich=null`.
- Таймауты 8с/20с — даже при полном офлайне ответ ≤ ~28с (в рамках ТЗ ≤30с).

### 12.1. Загрузчик Базы знаний (`compliance-kb.json`)

Отдельный воркфлоу (webhook `/compliance-kb`), **доступ любому вошедшему
пользователю** (проверяется валидный токен; роль `is_admin` больше не требуется —
по решению заказчика база знаний открыта всем сотрудникам). Действия: `kb_list`,
`kb_add_violation` (+ связи + эмбеддинг), `kb_add_measure`, `kb_add_requirement`,
`kb_add_norm` (нарезка текста → эмбеддинг каждого фрагмента → `kb_norms`),
`kb_delete`. Каждая запись векторизуется на сохранении (питает гибридный матч и
RAG). UI — вкладка «База знаний» в `compliance-agent.html` (видна всем вошедшим).

### 12.2. История

Действие `history` в `compliance.json` возвращает инциденты текущего юзера
(вкладка «История»); клик по записи открывает сохранённый отчёт.

## 13. FAQ и устранение проблем

**Карточка на дашборде красная.** Вебхук недоступен: воркфлоу не импортирован/не
активирован или n8n не запущен. `import-workflows.ps1 -Filter compliance` +
`activate-workflows.ps1`.

**«Сформировать меры» → `auth_required`.** Истёк токен — перелогиньтесь. Если
«бэкенд недоступен» — проверьте, что n8n запущен и вебхук активен; для показа
интерфейса используйте «Пример».

**Меры приходят только базовые + «не найдено».** Входящее нарушение не совпало ни
с одним `kb_violations`. Добавьте типовое нарушение и связи
([§6.2](#62-как-добавить-новое-типовое-нарушение-с-мерами)) **или** проверьте, что в
`title` нарушения есть те же ключевые слова, что во вводе пользователя
(матчинг идёт по значимым словам).

**«Утвердить» → ошибка.** Проверьте, что инцидент был создан (есть `incident_id`
в ответе analyze) и что токен валиден.

**Нужно меньше/больше «базовых» мер / другие сроки.** Заданы в код-ноде «Сборка
результата» (массивы `BASE` и `DL`).

**Сменили адрес n8n.** Поправьте `N8N_BASE` в `Agents/_config.js` — подхватят все агенты.

## 14. Чек-лист запуска

- [ ] `init-db.sql` накатан, `SELECT count(*) FROM kb_measures;` = 10 (или ваш сид).
- [ ] База знаний наполнена реальными нарушениями/мерами/связями организации.
- [ ] `import-workflows.ps1 -Filter compliance` → созданы **оба** воркфлоу (Поток + Загрузчик БЗ).
- [ ] Postgres-credential «Postgres» привязан в нодах.
- [ ] `activate-workflows.ps1` → вебхуки `compliance` и `compliance-kb` активны, карточка зелёная.
- [ ] Вкладка «База знаний» видна всем вошедшим (доступ открыт всем; роль админа не требуется).
- [ ] (Офис) `kb_violations` переиндексированы через `/embed` (семантический матч; иначе — текстовый фолбэк).
- [ ] `ping → pong` и `analyze без токена → auth_required` отвечают как ожидается.
- [ ] Полный цикл: «Сформировать меры» → «Утвердить» → инцидент `approved`, `kb_cases` вырос.
- [ ] Пользователю выдан [гайд пользователя](Compliance-User-Guide.md).

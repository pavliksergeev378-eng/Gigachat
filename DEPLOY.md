## Что нужно понимать перед началом

GigaChat-платформа состоит из двух частей:

1. **Backend** — n8n + PostgreSQL + GigaChat-LLM endpoint. Уже должны быть подняты где-то в LAN.
2. **Frontend** — статические HTML/JS/CSS файлы + Pyodide-бандл для math-агента. Это всё что в репозитории.

Frontend нужно как-то «раздать» пользователям. Два способа:

| Вариант | Что нужно | Кому подходит |
|---|---|---|
| **Portable Caddy** | Запустить локальный HTTP-сервер | Любой сценарий, особенно с math-агентом и тулзами |
| **Yandex с флагом** | Только Yandex Browser | Маленькая команда, без нужды в централизации |

**Важно:** math-агент использует Pyodide (Python в браузере через WebAssembly). Pyodide требует HTTP-origin для загрузки WASM. Через простой `file://`-доступ в Chromium-браузерах (включая Yandex) **не работает** — нужен либо HTTP-сервер (Вариант 1), либо специальный флаг для Yandex (Вариант 2).

## Вариант 1. Portable Caddy (рекомендуется)

В проекте уже лежит `caddy.exe` (40 МБ) — портативный HTTP-сервер без установки. Запускается одной командой через `GigaChat-Start.bat`, обслуживает все агенты на `http://localhost:8765`.

### Что понадобится

- Машина с Windows 10/11 (любая, где будет работать GigaChat)
- Доступ к офисной n8n по сети
- ZIP проекта с GitHub
- (Опционально) Python — только если хочешь использовать `python -m http.server` вместо Caddy

### Шаг 1. Подготовка машины

Если эта машина будет «сервером» для других пользователей — нужно открыть порт 8765 в firewall:

```powershell
New-NetFirewallRule -DisplayName "GigaChat HTTP" -Direction Inbound -LocalPort 8765 -Protocol TCP -Action Allow
```

Если только для тебя на твоей машине — этот шаг можно пропустить (localhost не блокируется).

### Шаг 2. Скачать и распаковать проект

1. Открой [https://github.com/Jorden-maker/GigaChat](https://github.com/Jorden-maker/GigaChat)
2. Нажми зелёную кнопку **Code → Download ZIP**
3. Распакуй в путь **без русских букв и пробелов**, например `C:\GigaChat\`

В папке должны быть:
```
C:\GigaChat\
├─ caddy.exe                ← портативный HTTP-сервер (40 МБ)
├─ Caddyfile                ← конфиг
├─ GigaChat-Start.bat       ← запуск (двойной клик)
├─ GigaChat-Platform.html   ← дашборд
├─ Agents\                  ← HTML страницы агентов
│  ├─ math-agent.html
│  ├─ test-pyodide.html     ← тест Pyodide без n8n и LLM
│  ├─ _config.js            ← конфигурация (МЕНЯТЬ В ШАГЕ 3)
│  ├─ _shared.js
│  └─ lib\pyodide\          ← Python-рантайм для math-агента (13 МБ, 9 файлов)
├─ Workflow\                ← JSON-файлы для импорта в n8n
└─ import-workflows.ps1     ← скрипт массового импорта workflow'ов
```

Проверь, что в `Agents\lib\pyodide\` лежат 9 файлов, особенно `pyodide.asm.wasm` (~10 МБ). Без них math-агент не заработает.

### Шаг 3. Настроить `_config.js`

Открой `C:\GigaChat\Agents\_config.js` в Блокноте. Внутри:

```js
window.GIGACHAT_CONFIG = {
  N8N_BASE: 'http://localhost:5678',
  PYODIDE_URL: 'lib/pyodide/pyodide.js',
  PYODIDE_INDEX_URL: 'lib/pyodide/'
};
```

**Замени `localhost:5678` на реальный адрес офисной n8n.** Варианты:

- `http://192.168.1.50:5678` — если n8n на другой машине в LAN с IP `192.168.1.50`
- `http://n8n.office.local:5678` — если есть внутренний DNS
- Оставить `localhost:5678` — если n8n запущен на этой же машине

**ВАЖНО:** оставь `PYODIDE_URL` и `PYODIDE_INDEX_URL` как есть — это локальные пути к Pyodide-бандлу.

### Шаг 4. Импорт workflow'ов в n8n

1. Открой офисную n8n в браузере (по адресу из `N8N_BASE`)
2. Перейди в **Settings → API → Create an API key**, скопируй ключ
3. (Только один раз!) В UI n8n создай credentials для PostgreSQL:
   - **Credentials → Add credential → Postgres**
   - Имя: **точно** `Postgres` (скрипт ищет по имени)
   - Заполни host/port/database/user/password офисной PostgreSQL
4. Открой `C:\GigaChat\import-workflows.ps1` в Блокноте, поправь:
   - `$apiKey = "..."` — вставь свежий API-ключ из шага 2
   - `$n8n = "http://localhost:5678"` — поменяй на реальный адрес офисной n8n
   - В `$credentialMapping.openAiApi.data`:
     - `url = "http://130.100.95.104:8810/v1"` — поменяй на URL офисного GigaChat-LLM
5. Запусти скрипт:
   ```powershell
   cd C:\GigaChat
   .\import-workflows.ps1
   ```
   Скрипт импортирует все 11 workflow'ов, создаст GigaChat credential автоматически и подменит ID в нодах.

### Шаг 5. Активировать workflow'ы

Импортированные workflow'ы — **неактивные** (draft). Их webhook URL не зарегистрирован. Без активации индикаторы на дашборде будут красные.

1. В n8n UI → **Workflows**
2. Открой каждый из них (`[GigaChat] Математический агент. Поток`, `[GigaChat] Чат...` и т.д.)
3. В правом верхнем углу — переключи тоггл **Active** (зелёный)

Всего 11 workflow'ов:
- Математический агент
- Чат-агент
- RAG-агент
- SQL-агент
- Prompt-engineer
- Роутер
- OCR + Поиск ФИО
- Загрузчик документов
- Извлечение текста
- Объединение таблиц Excel
- Загрузчик истории

### Шаг 6. Запустить Caddy

**Двойной клик** по `C:\GigaChat\GigaChat-Start.bat`.

Что должно произойти:
1. Откроется чёрное окно cmd с заголовком **«GigaChat Server»**
2. В окне напечатается шапка `============================ GigaChat Server URL: http://localhost:8765/ Close this window to stop the server. ============================`
3. Дальше пойдут логи Caddy (если включён INFO-уровень) или тишина (если ERROR — как у нас по дефолту)
4. Через ~3 секунды откроется браузер по умолчанию на `http://localhost:8765/`

**Окно cmd не закрывай!** Пока оно открыто — сервер работает. Закрыть = остановить.

### Шаг 7. Проверка работоспособности

Открой в браузере:

#### А. Дашборд

`http://localhost:8765/` — должен открыться GigaChat дашборд со всеми карточками агентов и инструментов.

Что проверить:
- Все 10 карточек (router, sql, rag, math, prompt, chat, ocr, docs, extract, merger-excel) должны быть **зелёные** («Онлайн»)
- Если красные — workflow не активирован в n8n (см. шаг 5) или `_config.js` указывает не на ту n8n (см. шаг 3)

#### Б. Тест Pyodide (изолированный)

`http://localhost:8765/Agents/test-pyodide.html` — нажми **«Запустить тесты»**.

12 тестов должны пройти за 2-3 секунды (первая загрузка Pyodide ~3 сек, дальше каждый тест 1-20 мс). Если все зелёные — Pyodide-инфраструктура для math-агента работает. Если красные — детали в колонке «Результат».

#### В. Math-агент end-to-end

`http://localhost:8765/Agents/math-agent.html` → создай сессию → задай:

- `2 + 2` → должен ответить `Ответ: 4` + объяснение от LLM
- `373 * 7373 * 272 / 232` → должен ответить точное число (тест Pyodide)
- `Площадь круга радиусом 5` → должен использовать `math.pi`

Если приходят ответы — **всё работает**.

### Раздача пользователям

Когда сервер развернут на одной машине в LAN, пользователям нужно только открыть URL:

```
http://<ip-сервера>:8765/
```

Способы раздачи:

#### Способ 1. Просто URL по почте/Teams

Самое простое — сообщить адрес. Пользователь добавит в закладки сам.

#### Способ 2. `.url`-ярлык

Создай файл `GigaChat.url` со следующим содержимым:

```ini
[InternetShortcut]
URL=http://192.168.1.50:8765/
IconIndex=0
```

(замени `192.168.1.50` на реальный IP)

Разошли пользователям. Они скопируют на рабочий стол — двойной клик откроет браузер по умолчанию на дашборд.

#### Способ 3. Корпоративное DNS-имя

Если в LAN настроен DNS, попроси сетевого админа добавить запись:

```
gigachat.office  →  192.168.1.50
```

Тогда URL будет `http://gigachat.office:8765/` — красивее и устойчивее к смене IP сервера.

Можно даже на 80 порт повесить — тогда `http://gigachat.office/`. Поменять в `Caddyfile` `:8765` на `:80` (нужны права на привязку, может потребоваться запуск Caddy от админа).

#### Способ 4. Hosts-файл на машинах пользователей

Если корпоративного DNS нет, можно через GPO Active Directory раскатать запись в `C:\Windows\System32\drivers\etc\hosts`:

```
192.168.1.50  gigachat
```

Тогда URL — `http://gigachat:8765/`.

## Вариант 2. Без сервера — через флаг `--allow-file-access-from-files`

**Когда выбирать:** маленькая команда (1-5 человек), нет желания поднимать сервер с Caddy, каждый пользователь со своей копией проекта.

**Как работает:** Chromium-браузер (Yandex/Chrome/Edge) запускается с особым флагом, который разрешает страницам с `file://` загружать другие локальные файлы через `fetch()`. Без флага Chromium блокирует это из соображений безопасности — а Pyodide именно `fetch` использует для загрузки WASM-файлов.

В проекте уже лежит готовый **`GigaChat-NoServer.bat`** — двойной клик и всё.

### Что понадобится

- Windows 10/11
- Yandex Browser, Chrome или Edge (любой Chromium)
- Папка с распакованным проектом (**без русских букв и пробелов** в пути!)
- n8n работает в LAN

### Шаг 1. Распаковать проект

Скачай ZIP с GitHub, распакуй в путь **без русских букв и пробелов**, например `C:\GigaChat\`.

⚠️ **Критично:** `file://`-URL с русскими символами или пробелами Chromium иногда не парсит. Не клади в `C:\Документы\` или `C:\Users\Иванов И.И.\`.

Проверь, что лежит `Agents\lib\pyodide\` с 9 файлами.

### Шаг 2. Настроить `_config.js`

Открой `C:\GigaChat\Agents\_config.js` и замени `localhost:5678` на реальный адрес офисной n8n. Например:

```js
N8N_BASE: 'http://192.168.1.50:5678',
```

### Шаг 3. Двойной клик по `GigaChat-NoServer.bat`

`.bat` сам ищет браузер (Yandex → Chrome → Edge) и запускает с правильными флагами:

- `--allow-file-access-from-files` — разрешает `fetch()` к локальным файлам
- `--user-data-dir="%USERPROFILE%\GigaChatBrowserProfile"` — **изолированный** профиль (без твоих закладок и истории — это критично для security)

Откроется отдельное окно браузера с дашбордом GigaChat.

### Шаг 4. Проверка (важно сделать СРАЗУ)

Открыл окно? Отлично. Теперь сразу:

1. **F12 → вкладка Console** — нет ли красных ошибок?
   - `Access to fetch ... blocked by CORS policy` → CORS-проблема, см. раздел [Проблемы](#-высокая-вероятность)
   - `Failed to fetch ... ERR_FAILED` → `_config.js` указывает не на ту n8n
   - Чисто → идём дальше
2. **Дашборд → индикаторы зелёные?**
   - Все зелёные → пинги webhook'ов проходят, всё ок
   - Все красные → workflow'ы не активированы в n8n ИЛИ CORS-проблема
3. **Открой `file:///C:/GigaChat/Agents/test-pyodide.html`** → нажми «Запустить тесты»
   - Все 14 зелёные → Pyodide-цепочка через file:// + Worker работает
   - Если какие-то красные → раскрой детали в колонке «Результат»
4. **Клик «Математика»** на дашборде → задай `2 + 2`
   - Получил `Ответ: 4` + пояснение → **полная цепочка работает**, NoServer-вариант годен для офиса

### Что увидит конечный пользователь

Этот же `.bat` раздать каждому пользователю (положить ярлык на рабочий стол). Двойной клик → отдельное окно браузера сразу с дашбордом. Никаких терминалов.

### Минусы и риски этого подхода

- **На каждой машине нужен свой `.bat` или ярлык** — централизованной раскатки нет, флешка/общий диск/email
- **Изолированный профиль = «голый» браузер без твоих обычных закладок и расширений** — пользователь может растеряться при первом запуске
- **Security caveat.** В этом окне НЕ открывать посторонние сайты — внутри изолированного профиля security ослаблена. Изоляция через `--user-data-dir` снижает риск, но не на 100%
- **Yandex/Chrome иногда обновляется и меняет поведение флагов** — раз в полгода может потребоваться пересоздать профиль (удалить `%USERPROFILE%\GigaChatBrowserProfile`) или поправить флаг
- **Папка `GigaChatBrowserProfile` растёт** (~50-100 МБ) — раз в полгода почистить
- **GPO в корпсети может игнорировать флаги** — если IT настроил жёсткие политики на Yandex/Chrome, флаги могут не сработать. Симптом: открывается ОБЫЧНОЕ окно браузера с твоими закладками вместо изолированного. В таком случае — переходи на Вариант 1 (Caddy)

### Кастомизация (если нужно)

`.bat` лежит в корне проекта, можно открыть в Блокноте и поправить:
- Порядок поиска браузеров (по умолчанию: Yandex → Chrome → Edge)
- Имя/расположение профиля (по умолчанию: `%USERPROFILE%\GigaChatBrowserProfile`)
- Дополнительные флаги Chromium

## Какой вариант выбрать в офисе

| Критерий | Вариант 1 (Caddy) | Вариант 2 (NoServer) |
|---|---|---|
| Нужен HTTP-сервер | Да (фоновый Caddy) | Нет |
| Видимое окно cmd в Taskbar | Да | Нет |
| Подходит для централизованной раскатки | Да (один сервер на LAN) | Нет (каждому свой пакет) |
| Зависит от типа браузера | Любой (даже Firefox) | Только Chromium |
| Зависит от корпоративных GPO | Минимально | Может быть проблемой |
| Размер дистрибутива | +40 МБ (caddy.exe) | базовый |
| Сложность установки | Двойной клик `.bat` | Двойной клик `.bat` |

**Стратегия с минимальным риском:** в офисе сначала пробуй **Вариант 2** (`NoServer.bat`). Если работает — отлично, проще. Если упирается в CORS / GPO / другую блокировку — переходи на **Вариант 1** (`Start.bat`). Оба `.bat` лежат рядом в проекте, готовы.

**Преимущество Caddy именно для офиса:** origin становится `http://localhost:8765` вместо `file://`. Большинство корпоративных security-политик нацелены на `file://`-доступ, локальный HTTP не трогают. Поэтому Caddy более устойчив к жёстким корпоративным настройкам.

## Ручная настройка ярлыка (без `.bat`)

Если `GigaChat-NoServer.bat` по какой-то причине не сработал, можно создать ярлык вручную:

1. ПКМ на рабочем столе → **Создать → Ярлык**
2. В поле «Расположение объекта» вставь (одной длинной строкой):

```
"C:\Users\<имя>\AppData\Local\Yandex\YandexBrowser\Application\browser.exe" --allow-file-access-from-files --user-data-dir="C:\GigaChatProfile" "file:///C:/GigaChat/GigaChat-Platform.html"
```

Подставь свой `<имя>` и проверь путь к `browser.exe`. Путь к проекту тоже подстроить под себя.

3. Нажми **Далее** → имя ярлыка: **GigaChat** → **Готово**
4. Опционально: ПКМ на ярлыке → **Свойства → Сменить значок**

Что делает каждый флаг:

| Флаг | Зачем |
|---|---|
| `--allow-file-access-from-files` | Разрешает `fetch()` от `file://` к локальным файлам. Без него Pyodide не загрузится. |
| `--user-data-dir="..."` | **Критично для security.** Изолирует в отдельный профиль Chromium. Без него флаг бы применился ко всем твоим обычным вкладкам — любой подозрительный сайт получил бы доступ к локальным файлам. |
| `"file:///C:/.../GigaChat-Platform.html"` | URL дашборда для немедленного открытия. |

## Что увидит конечный пользователь

После любого из двух вариантов развёртывания у пользователя такой workflow:

1. Двойной клик по ярлыку/иконке («GigaChat» на рабочем столе)
2. Открывается браузер с дашбордом — плитки агентов и инструментов с индикаторами статуса
3. Кликает по плитке нужного агента — открывается интерфейс агента
4. Создаёт сессию, отправляет запрос, получает ответ

**Никакого PowerShell, никаких терминалов, никаких ручных команд.** Только клики мышью.

## Возможные проблемы и решения

### 🔴 Высокая вероятность

#### 1. Антивирус блокирует `caddy.exe`

Это **самая частая проблема** в корпоративной среде. Корпоративные антивирусы (Kaspersky, Symantec, ESET, Defender в strict-режиме) ругаются на:
- Unsigned бинарь (нет цифровой подписи разработчика)
- 40 МБ файл, скачанный из интернета
- Открывает сетевой порт, делает HTTP-сервер

Что может сделать антивирус:
- Тихо заблокировать запуск
- Поместить файл в карантин (исчезает с диска)
- Показать popup пользователю
- Отключить от сети

**Симптом:** окно `GigaChat-Start.bat` открывается, печатает шапку, потом мгновенно «Caddy stopped» и пауза. Или вообще закрывается с «caddy.exe not found».

**Что проверить:**
- В антивирусе посмотреть «Карантин» / «Заблокированные файлы» — файл `C:\GigaChat\caddy.exe` будет там
- Открыть журнал событий: `Get-EventLog -LogName Application -Newest 20 | Where-Object { $_.Message -match 'caddy' }`

**Как чинить:**
- Добавить `caddy.exe` или папку `C:\GigaChat\` в исключения антивируса (нужны права админа на этой машине)
- Попросить IT добавить в whitelist на корпоративном уровне
- Подписать `caddy.exe` корпоративным сертификатом (если IT может)
- Скачать заново, если файл удалён из карантина

#### 2. `_config.js` указывает на `localhost:5678`, а n8n на другой машине

**Симптом:** дашборд открылся, но все 10 индикаторов красные («Офлайн»).

**Что проверить:** F12 в браузере → вкладка **Network** → попробуй кликнуть карточку → увидишь fetch на `localhost:5678` который падает с `ERR_CONNECTION_REFUSED` или `ERR_NETWORK`.

**Как чинить:** в `Agents\_config.js` заменить `localhost:5678` на реальный адрес n8n. Например:

```js
N8N_BASE: 'http://192.168.1.50:5678',
```

После сохранения — обновить страницу (Ctrl+F5).

#### 3. Workflow'ы в n8n не активированы после импорта

Скрипт `import-workflows.ps1` создаёт workflow'ы как draft (неактивные). Webhook URL'ы у неактивных workflow'ов n8n **не регистрирует** — любой POST вернёт 404.

**Симптом:** дашборд открыт, некоторые/все индикаторы красные. В F12 → Network видно, что пинги возвращают **404** «webhook is not registered».

**Как проверить:** в n8n UI → Workflows → колонка «Active» — у активных будет зелёный кружок.

**Как чинить:** открыть каждый workflow → переключить тоггл **Active** в верхнем правом углу.

Всего 11 workflow'ов. Можно массово: на странице Workflows выделить все → Bulk Activate (если в твоей версии n8n есть эта опция).

### 🟡 Средняя вероятность

#### 4. n8n не отдаёт CORS-заголовки на webhook'и

Если в офисной n8n включён strict CORS (через env `N8N_CORS_ENABLED` или похожее), браузер блокирует fetch с другого origin.

**Симптом:** в F12 → Console красные ошибки:
```
Access to fetch at 'http://192.168.1.50:5678/webhook/math?ping=1' from origin 'http://localhost:8765' has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present on the requested resource.
```

Для **Варианта 2 (NoServer.bat)** особенно: Chromium для `file://` страниц шлёт `Origin: null` — если n8n CORS-whitelist не включает `null`, все запросы будут блокироваться.

**Как чинить:** в env n8n (`.env` файл или docker run -e) добавить:

```
N8N_CORS_ALLOW_ORIGIN=*
```

Или явно указать домен Caddy:

```
N8N_CORS_ALLOW_ORIGIN=http://localhost:8765,http://192.168.1.50:8765
```

После — перезапустить n8n.

#### 5. Браузер по умолчанию не Yandex

`start "" "http://..."` в `GigaChat-Start.bat` открывает в браузере по умолчанию. Если корп.политика поставила Edge — откроется Edge. Не баг, но неожиданный UX.

**Как чинить:**

Способ 1 — поменять браузер по умолчанию в Windows (Settings → Apps → Default apps → Web browser).

Способ 2 — в `GigaChat-Start.bat` заменить строку:

```batch
start "" "http://localhost:8765/"
```

на явный путь к Yandex:

```batch
start "" "%LOCALAPPDATA%\Yandex\YandexBrowser\Application\browser.exe" "http://localhost:8765/"
```

#### 6. (NoServer) GPO блокирует флаги Chromium

В жёстко-настроенных корпсетях IT через групповые политики (GPO) для Yandex/Chrome/Edge может:
- Запретить запуск с командными флагами (`--allow-file-access-from-files`, `--user-data-dir`)
- Принудительно использовать единственный профиль (игнорировать `--user-data-dir`)
- Отключить любые «developer» флаги

**Симптом:** двойной клик по `GigaChat-NoServer.bat` → открывается **обычное** окно Yandex с твоими закладками и историей вместо изолированного. Pyodide не загружается (на дашборде идут CORS-ошибки в Console).

**Что проверить:** в открывшемся окне в адресной строке набрать `chrome://policy/` (или `browser://policy/` для Yandex) — увидишь список действующих корпоративных политик.

**Как чинить:** только через IT — попросить добавить исключение для GigaChat в политику. Если IT не пойдёт навстречу — **переходи на Вариант 1 (`Start.bat` с Caddy)**. Caddy создаёт `http://localhost:8765` origin вместо `file://`, и большинство корпоративных политик нацелены именно на `file://`-доступ.

#### 7. (NoServer) Путь с русскими буквами или пробелами не парсится

Chromium иногда не парсит `file://`-URL с русскими символами или пробелами. Если проект распакован в `C:\Документы\GigaChat\` или `C:\Users\Иванов И.И.\GigaChat\` — окно откроется, но страница не загрузится.

**Симптом:** браузер открыт, в адресной строке `file:///C:/...` с искажёнными русскими символами, страница пустая или 404.

**Как чинить:** перенести проект в путь без русских букв и пробелов (`C:\GigaChat\`, `D:\Apps\GigaChat\`).

#### 8. Порт 8765 уже занят

Какой-то другой процесс уже слушает 8765 (редко, но возможно).

**Симптом:** в окне `GigaChat-Start.bat` сразу после запуска:

```
Error: listen tcp :8765: bind: address already in use
Caddy stopped.
```

**Как проверить:**

```powershell
netstat -ano | findstr :8765
```

Покажет PID процесса. Дальше:

```powershell
Get-Process -Id <PID>
```

— увидишь имя программы.

**Как чинить:** в `Caddyfile` поменять `:8765` на свободный порт (например `:9876`). Не забыть поправить URL в `GigaChat-Start.bat`:

```batch
start "" "http://localhost:9876/"
```

### 🟢 Низкая вероятность

#### 9. (Caddy) Путь с русскими буквами/пробелами ломает `%~dp0`

Если развернёшь в `C:\Документы пользователя\Мои файлы\GigaChat\` или `\\fileserver\Общие документы\GigaChat\` — `cmd.exe` иногда косячит парсинг `%~dp0` в `GigaChat-Start.bat`.

**Симптом:** «caddy.exe not found» хотя файл там есть.

**Как чинить:** переместить проект в путь без русских букв и пробелов, например `C:\GigaChat\`. (То же касается NoServer-варианта — см. #7.)

#### 10. OneDrive синхронизирует папку проекта

Если пользователь распакует ZIP в `C:\Users\<имя>\OneDrive\...\GigaChat\`, OneDrive может:
- Заблокировать `.exe` от исполнения
- Заменить файлы на placeholder («скачать по требованию»)
- Конфликтовать с открытыми файлами Caddy

**Как чинить:** не клади проект в OneDrive-синхронизированную папку. Используй `C:\GigaChat\` или `D:\GigaChat\`.

#### 11. Корпоративная политика запрета `.bat`/`.exe` из user folders

Жёстко-настроенные корпоративные машины могут запрещать запуск любых исполняемых файлов из `C:\Users\` (Software Restriction Policy или AppLocker).

**Симптом:** двойной клик по `.bat` — ничего не происходит, или Windows показывает «Эта программа заблокирована групповой политикой».

**Как чинить:** только через IT — переместить проект в `C:\Program Files\GigaChat\` (требует админа для установки) и добавить в whitelist политики. Если IT не пойдёт навстречу — переходить на desktop-приложение (Electron) с подписью.

#### 12. Несколько RDP-пользователей на одной машине

Если «сервер» — это Windows Terminal Server, и каждый пользователь запускает свой `GigaChat-Start.bat`, второй получит «порт 8765 занят» (первый уже взял).

**Как чинить:** либо один общий Caddy под системным аккаунтом (запустить как Windows-сервис), либо каждому свой порт.

#### 13. lib/pyodide/ отсутствует в проекте

Если скачали старый ZIP (до того как Pyodide добавили в репо) или папка случайно удалена антивирусом.

**Симптом:** math-agent открывается, но при первом запросе ошибка «Не удалось загрузить Python-рантайм».

**Как проверить:**

```powershell
Get-ChildItem C:\GigaChat\Agents\lib\pyodide\
```

Должно быть 9 файлов. Особенно важны `pyodide.asm.wasm` (~10 МБ), `python_stdlib.zip` (~2 МБ), `pyodide.js`.

**Как чинить:** скачать свежий ZIP с GitHub.

#### 14. MIME-тип для WASM неверный

Caddy раздаёт `.wasm` файлы. Если Caddyfile повреждён или потёрли правило `@wasm`, Pyodide получает `application/octet-stream` вместо `application/wasm`.

**Симптом:** в F12 → Console: `Failed to compile WebAssembly module` или `Refused to execute script... MIME type ('application/octet-stream') is not executable`.

**Как проверить:** в браузере открой напрямую `http://localhost:8765/Agents/lib/pyodide/pyodide.asm.wasm`. Браузер должен начать скачивать (~10 МБ). В DevTools посмотри Headers → Content-Type — должно быть `application/wasm`.

**Как чинить:** проверить `Caddyfile` — должно быть правило:

```
@wasm path *.wasm
header @wasm Content-Type application/wasm
```

Если нет — добавить и перезапустить Caddy (закрыть окно, запустить заново).

## Чеклист диагностики «когда что-то не работает»

Иди сверху вниз — это покрывает 95% проблем.

### 1. Окно `GigaChat-Start.bat` мгновенно закрывается

- Антивирус блокирует `caddy.exe` → см. **Проблема 1**
- Или путь к проекту с русскими буквами → см. **Проблема 7**

### 2. Окно открывается, но через секунду «Caddy stopped»

В окне `cmd` будет причина:
- `bind: address already in use` → порт занят → **Проблема 6**
- `permission denied` → нужны права админа или порт <1024
- Что-то про `Caddyfile` → синтаксическая ошибка в конфиге

### 3. Caddy работает, дашборд открылся, но индикаторы красные

Проверь по порядку:
- В `_config.js` правильный адрес n8n? → **Проблема 2**
- Открой адрес n8n в браузере напрямую — UI открывается? Если нет, n8n не работает в LAN.
- В n8n UI workflow'ы активированы? → **Проблема 3**
- В F12 → Console есть CORS-ошибки? → **Проблема 4**

### 4. Индикаторы зелёные, но math падает на запросе

LLM-эндпоинт. В n8n открой workflow «Математический агент» → вкладка **Executions** → последний execution → найди ноду «Генерация кода» → посмотри ошибку. Обычно одно из:
- LLM URL неправильный в credential
- LLM сервер недоступен по сети
- Таймаут (LLM очень медленный)

### 5. Math работает на простых задачах, но падает на сложных

- В окне math-agent появилась ошибка «Python timeout» → код LLM написал бесконечный цикл, нужен релоад страницы
- В ответе «Не удалось загрузить Python-рантайм» → проблема с Pyodide → **Проблема 11/12**

### 6. Math на одном вопросе работает, на другом — нет

LLM непредсказуем. Открой F12 → Network → найди запрос `/webhook/math` → посмотри ответ. Там `code` — что LLM сгенерировал. Часто видно `import sympy` или подобное — значит LLM ошибся (sympy не доступен в нашем bundle). Workflow должен такое отлавливать в проверке безопасности, но если не отловил — баг.

## CORS — что это и как чинить (подробно)

**CORS** = Cross-Origin Resource Sharing. Это **защитный механизм браузера**, не позволяющий зловредному сайту дёргать API других сайтов от твоего имени. В GigaChat-офисном деплое CORS — самая частая причина «индикаторы красные на чужом ПК, а на моём зелёные».

### Когда CORS вообще релевантен

`Origin` = протокол + хост + порт. Например:
- `http://130.100.94.119:8765` — origin A
- `http://130.100.94.119:5678` — origin B (порт отличается → ДРУГОЙ origin)
- `http://localhost:8765` и `http://130.100.94.119:8765` — разные (хост отличается)

Если JS на странице делает `fetch()` к URL **с другим origin** — это cross-origin запрос. Сервер должен явно разрешить через заголовок `Access-Control-Allow-Origin`, иначе браузер **молча блокирует ответ** (промис в JS отклоняется, в Console — ошибка).

### Когда у тебя точно будет CORS

Любой из этих случаев:

1. **Caddy на твоём ПК + n8n на другом сервере** (типовой офисный сетап):
   - Страница: `http://130.100.94.119:8765` ← Caddy
   - Webhook'и: `http://<n8n-server-ip>:5678` ← n8n
   - Разные хосты → cross-origin

2. **Caddy и n8n на одной машине, но разные порты:**
   - Страница: `http://localhost:8765` ← Caddy
   - Webhook'и: `http://localhost:5678` ← n8n
   - Разные порты → cross-origin

3. **Открытие через `file://` (GigaChat-NoServer.bat вариант):**
   - Страница: `file:///C:/GigaChat/...`
   - Webhook'и: `http://<n8n-ip>:5678`
   - Chromium для file:// шлёт `Origin: null` → cross-origin с особенностями

**Когда CORS НЕ нужен:** если страница и n8n на одном origin (один хост + один порт). В GigaChat-сетапе это редко.

### Pre-flight (двойной запрос)

Когда браузер делает POST с `Content-Type: application/json` (это «непростой» запрос по спецификации), он **сначала** делает пробный OPTIONS-запрос («можно?»). Только если сервер ответил с правильными CORS-заголовками, идёт настоящий POST.

```
Браузер → n8n:    OPTIONS /webhook/chat
                  Origin: http://130.100.94.119:8765
                  Access-Control-Request-Method: POST
                  Access-Control-Request-Headers: Content-Type

n8n    → Браузер: 200 OK
                  Access-Control-Allow-Origin: *
                  Access-Control-Allow-Methods: POST
                  Access-Control-Allow-Headers: Content-Type

Браузер → n8n:    POST /webhook/chat (настоящий запрос)
                  ...
```

Если OPTIONS не отвечает или возвращает не те заголовки, до POST дело не доходит.

### Как точно понять что у тебя CORS-проблема

#### Шаг 1. Открой F12 → вкладка Console

Конкретные сообщения, означающие CORS:

```
Access to fetch at 'http://<n8n-ip>:5678/webhook/chat?ping=1'
from origin 'http://130.100.94.119:8765' has been blocked by CORS policy:
No 'Access-Control-Allow-Origin' header is present on the requested resource.
```

→ **n8n не отдаёт CORS-заголовков совсем**. Самый частый случай.

```
Access to fetch at '...' from origin '...' has been blocked by CORS policy:
The 'Access-Control-Allow-Origin' header has a value 'http://localhost:8765'
that is not equal to the supplied origin.
```

→ **n8n отдаёт CORS только на конкретный origin**, и текущий не в whitelist.

```
Access to fetch at '...' has been blocked by CORS policy:
Response to preflight request doesn't pass access control check
```

→ **OPTIONS-запрос (preflight) не прошёл**. n8n не отвечает на OPTIONS или не присылает нужных заголовков.

#### Шаг 2. F12 → вкладка Network → клик на любой ping-запрос к webhook

В правой панели появится разбивка. Смотри:

- **Status:** должен быть `200`. Если `0` или `(failed)` — браузер не получил ответ вообще (заблокировал).
- **Headers → Response Headers**: ищи строку `access-control-allow-origin`.
  - Если её нет → CORS не настроен у n8n
  - Если есть со значением `*` → разрешено всем
  - Если есть с конкретным URL → разрешено только этому origin

### Как чинить — на стороне n8n (не на твоём ПК!)

CORS-настройка делается **на машине где запущен n8n**, не на твоём ПК с Caddy. Если у тебя нет доступа к этой машине — попроси админа n8n.

n8n управляется через переменные окружения. Нужная переменная — `N8N_CORS_ALLOW_ORIGIN`.

#### Универсальное решение (для офисного MVP)

Разрешить запросы со всех origin'ов:

```
N8N_CORS_ALLOW_ORIGIN=*
```

В корпоративной LAN, отделённой от интернета, это безопасно — webhook'и n8n и так доступны любой машине в LAN (нет аутентификации).

#### Более строгое решение (для прода)

Указать конкретные origin'ы через запятую:

```
N8N_CORS_ALLOW_ORIGIN=http://130.100.94.119:8765,http://gigachat.office:8765,http://localhost:8765
```

Менее удобно — каждый раз когда меняется IP или добавляется хост, надо обновлять.

### Как применить env-переменную к n8n

Зависит от того, как n8n запущен на твоём сервере.

#### Вариант A: n8n в Docker

```bash
docker stop n8n
docker rm n8n
docker run -d --name n8n -p 5678:5678 \
  --restart=unless-stopped \
  -e N8N_CORS_ALLOW_ORIGIN='*' \
  -v n8n_data:/home/node/.n8n \
  n8nio/n8n
```

Или если используется `docker-compose.yml`, добавить в секцию environment:

```yaml
services:
  n8n:
    image: n8nio/n8n
    environment:
      - N8N_CORS_ALLOW_ORIGIN=*
      # ... остальные env ...
```

И перезапустить: `docker-compose up -d`.

#### Вариант B: n8n как Windows-сервис (через nssm или PM2)

1. Найди скрипт запуска или конфиг сервиса (обычно `.bat` или `.json` рядом с n8n)
2. Добавь установку env перед запуском:
   ```batch
   set N8N_CORS_ALLOW_ORIGIN=*
   n8n start
   ```
3. Перезапусти сервис:
   ```powershell
   Stop-Service n8n
   Start-Service n8n
   ```

#### Вариант C: n8n через npm/Node.js напрямую

Найди `.env` файл в директории установки n8n (или в `~/.n8n/`). Добавь строку:

```
N8N_CORS_ALLOW_ORIGIN=*
```

Перезапусти n8n (Ctrl+C → `n8n start` снова).

#### Вариант D: n8n.io Cloud

CORS управляется через интерфейс n8n Cloud, нужно искать в настройках workspace. Менее вероятный сценарий для офисного развёртывания, обычно офисы хостят сами.

### Как проверить что починилось

После перезапуска n8n:

1. **Локально с твоего ПК:**
   - Открой `http://localhost:8765/` (или твой Caddy-URL)
   - Все индикаторы должны быть зелёные

2. **С другого ПК в LAN:**
   - Открой `http://130.100.94.119:8765/` (твой IP)
   - Все индикаторы должны стать зелёные

3. **В F12 → Network → любой ping-запрос → Response Headers:**
   - Должна быть строка `access-control-allow-origin: *` (или твой origin)

### Подводные камни

#### Pre-flight кэшируется

После настройки CORS, **закрой и снова открой вкладку браузера** — иначе старый preflight в кэше может продолжать блокировать. Или используй Ctrl+Shift+R (хард-перезагрузка).

#### n8n версия имеет значение

В очень старых версиях n8n (до 0.150 или около того) переменная называлась иначе или была глючная. Если N8N_CORS_ALLOW_ORIGIN не работает — обнови n8n до актуальной.

#### Webhook без авторизации

CORS — это только защита браузера. **Любая машина в LAN может сделать `curl http://n8n-ip:5678/webhook/chat`** без CORS-проблем (curl не браузер). CORS не заменяет аутентификацию. Если webhook'и должны быть закрыты от чужих машин — нужна отдельная авторизация (Basic Auth, JWT, IP whitelist в reverse-proxy).

#### Если N8N_CORS_ALLOW_ORIGIN=* не помогает

- Проверь что n8n РЕАЛЬНО перезапустился (env-переменная применяется только при старте). `docker ps` или `Get-Service n8n` или `ps aux | grep n8n`.
- Проверь что в Response Headers (F12) появилась строка — если не появилась, env не применилась.
- Возможно за n8n стоит reverse-proxy (nginx/traefik), который режет CORS-заголовки. Проверь конфиг proxy.

### Краткая шпаргалка диагностики

```
Симптом                          | Причина             | Действие
─────────────────────────────────|─────────────────────|─────────────────────
F12 Console: "blocked by CORS"   | N8N не отдаёт CORS  | N8N_CORS_ALLOW_ORIGIN=*
F12 Network status: 0 / failed   | preflight упал       | то же
Response Headers: нет CORS строк | N8N не отдаёт CORS  | то же
Response: ALL-O: localhost:8765  | whitelist узкий     | N8N_CORS_ALLOW_ORIGIN=*
Локально зелёные, в LAN красные  | whitelist на лок-IP | то же
Всё всегда красные               | n8n не запущен      | проверить процесс
```

## Настройка OCR и embedding-серверов через IP

Этот раздел нужен, если **n8n работает на одном ПК**, а **OCR-сервер и embedding-сервер — на другом** (твой ПК в офисе). n8n со своей машины должен достучаться до твоего ПК по сети — через IP.

### Архитектура

```
[ПК Васи / Пети / ...]              [n8n-сервер]                  [Твой ПК в офисе]
браузер с дашбордом   ──HTTP──>     n8n :5678   ──HTTP──>         ├─ OCR-server :8055
http://<N8N_HOST>:5678                                            └─ embedding :8001
```

Три переменные, которые надо подставить (см. также: коммит-сообщения проекта про плейсхолдеры):

| Плейсхолдер | Что это | Где подставляется |
|---|---|---|
| `<N8N_HOST>` | IP/hostname ПК с n8n | `Agents/_config.js` |
| `<OCR_HOST>` | IP/hostname твоего ПК (где OCR-сервер) | 4 workflow'а в n8n (нода `OCR (text extraction)`) |
| `<EMBEDDING_HOST>` | IP/hostname твоего ПК (где embedding) | 2 workflow'а: `document-loader` и `rag-agent` (ноды эмбеддинга) |

`<OCR_HOST>` и `<EMBEDDING_HOST>` обычно один и тот же IP — оба сервера стоят на твоём ПК.

### Шаг 1. Установить серверы

Если ещё не сделано:

- **OCR-сервер** — пошагово в [`ocr-server/README.md`](ocr-server/README.md). Качается с GitHub Releases, ставится в офлайне. Слушает порт `8055`.
- **Embedding-сервер** — пошагово в [`embedding-server/README.md`](embedding-server/README.md). Аналогично. Слушает порт `8001`.

Оба сервера по умолчанию слушают на `0.0.0.0` — это значит, что они доступны и локально (`localhost`), и из LAN (по IP). Менять ничего не нужно.

### Шаг 2. Узнать IP своего ПК

В PowerShell или cmd:

```cmd
ipconfig
```

Найди раздел активного сетевого адаптера (обычно «Ethernet adapter» или «Wireless LAN adapter»). Ищи строку:

```
IPv4-адрес. . . . . . . . . . . . : 192.168.1.50
```

Это твой IP в LAN. Запиши его.

**Какие IP бывают:**
- `192.168.x.x` — самый частый офисный IP, **приватный**, безопасный.
- `10.x.x.x` — тоже приватный, корпоративные сети.
- `172.16.x.x` – `172.31.x.x` — тоже приватный.
- Что-то ещё — возможно публичный, **не светить наружу**.

### Шаг 3. Зафиксировать IP (статический или DHCP reservation)

Если IP сейчас выдаётся динамически (DHCP), при перезагрузке роутера он может смениться — тогда n8n перестанет находить OCR/embedding. Варианты:

- **DHCP reservation в роутере** (рекомендуется): привязать MAC-адрес твоего ПК к этому IP. Открыть админку роутера → DHCP → Reservation → добавить.
- **Статический IP в Windows**: Параметры → Сеть → Адаптер → IP-конфигурация → Изменить → Вручную → IPv4 → ввести IP, маску, шлюз, DNS. Менее гибко, чем reservation.

### Шаг 4. Открыть порты в firewall на твоём ПК

n8n с другого ПК должен пройти через Windows Firewall к OCR (8055) и embedding (8001). От имени **админа** в PowerShell:

```powershell
New-NetFirewallRule -DisplayName "OCR LAN inbound" -Direction Inbound -LocalPort 8055 -Protocol TCP -Action Allow
New-NetFirewallRule -DisplayName "Embedding LAN inbound" -Direction Inbound -LocalPort 8001 -Protocol TCP -Action Allow
```

Если в офисе используется корпоративный антивирус с собственным firewall — открыть те же порты и в нём.

### Шаг 5. Запустить серверы

На твоём ПК — двойной клик по:

- `C:\ocr-server\start.bat` — откроется окно с логами OCR
- `C:\embedding-server\start.bat` — окно с логами embedding

Окна **не закрывать** — пока они открыты, серверы работают.

Проверка локально (на том же ПК):

```powershell
curl http://localhost:8055/status
curl http://localhost:8001/health
```

Должны отдать `{"status":"ok",...}`.

Проверка из LAN (с любого другого ПК офиса, подставь свой IP):

```powershell
curl http://192.168.1.50:8055/status
curl http://192.168.1.50:8001/health
```

Если первый отвечает, а второй нет — проблема в firewall на конкретный порт. Если оба не отвечают из LAN — firewall режет всё, или сервер слушает только `127.0.0.1`.

### Шаг 6. Подставить IP в `Agents/_config.js`

На ПК, где лежит фронт (Caddy или у каждого юзера локально):

```js
// Agents/_config.js
window.GIGACHAT_CONFIG = {
  N8N_BASE: 'http://192.168.1.10:5678',  // ← IP n8n-сервера, не твоего!
  PYODIDE_URL: 'lib/pyodide/pyodide.js',
  PYODIDE_INDEX_URL: 'lib/pyodide/'
};
```

⚠️ Сюда идёт **IP n8n-сервера**, не твоего. Браузер ходит на n8n, n8n ходит на OCR/embedding — разные адреса.

### Шаг 7. Подставить IP в workflow'ах n8n

Два способа, выбрать один:

**Способ A — править JSON-файлы перед импортом (быстро):**

В папке `Workflow/` найди и замени через любой текстовый редактор:
- `<OCR_HOST>` → `192.168.1.50` (твой IP) — 4 файла
- `<EMBEDDING_HOST>` → `192.168.1.50` (твой IP) — 2 файла

Файлы:
- `Workflow/organization-appeal.json` — OCR
- `Workflow/document-loader.json` — OCR + embedding
- `Workflow/OCR + Поиск ФИО.json` — OCR
- `Workflow/text-extractor.json` — OCR
- `Workflow/rag-agent.json` — embedding

Потом импортируй через `import-workflows.ps1` или вручную в n8n.

⚠️ **После замены не делай `git add` этих файлов** — реальный IP не должен попасть в публичный репозиторий.

**Способ B — править прямо в n8n UI после импорта (чище):**

1. Импортируй workflow'ы как есть (с плейсхолдерами).
2. Открой workflow в n8n.
3. Кликни на ноду `OCR (text extraction)` или `Эмбеддинг`.
4. В поле URL замени `<OCR_HOST>`/`<EMBEDDING_HOST>` на реальный IP.
5. Save.

Этот способ оставляет исходники в репозитории чистыми — изменения живут только в БД n8n.

### Шаг 8. Активировать workflow'ы

В n8n каждый workflow → тумблер **Active** → ON.

### Шаг 9. Проверка из дашборда

1. Открой дашборд в браузере (`GigaChat-Platform.html` через Caddy или напрямую).
2. Все 11 status-точек должны загореться зелёным.
3. Если красные — F12 → Network → попробуй кликнуть карточку и посмотри, на какой URL фронт стучится:
   - Если `localhost:5678` — значит `_config.js` не подхватился, проверь шаг 6.
   - Если правильный IP, но `ERR_CONNECTION_REFUSED` — n8n не отвечает или firewall на n8n-сервере закрыт.
4. Зайди в `Алгоритмы → Организация обращения` → загрузи `Tests/OrgAppeal/01-ivanov-full-success.pdf` → должен вернуться итог. Если упадёт на «OCR не распознал» — проверь шаг 5 (OCR-сервер) и шаг 4 (firewall).

### Типичные проблемы

| Симптом | Где смотреть |
|---|---|
| Все карточки красные | `_config.js` не подхватился. F12 → Console → `GigaChat.config.N8N_BASE`. Должен быть твой IP, не `<N8N_HOST>`. |
| Workflow выполняется, но падает на ноде OCR с `ECONNREFUSED` | OCR-сервер не запущен (шаг 5), или firewall на твоём ПК закрыт (шаг 4), или в workflow остался плейсхолдер `<OCR_HOST>` (шаг 7). |
| `ETIMEDOUT` вместо `ECONNREFUSED` | Сервер запущен, но firewall дропает пакет молча. Проверь правило firewall (шаг 4). |
| Локально PDF обрабатывается, по IP — нет | Сервер слушает `127.0.0.1` вместо `0.0.0.0`. Проверь `start.bat`: переменная `OCR_HOST` (или `EMBED_HOST`) должна быть `0.0.0.0` или не задана (default). |
| IP сменился после перезагрузки | Шаг 3 не выполнен. DHCP выдал новый адрес. Зафиксируй IP. |
| Браузер: `Mixed Content blocked` | Дашборд открыт по `https://`, а n8n по `http://`. Либо открой дашборд по `http`, либо поставь TLS-сертификат на n8n. |

### Полезные команды для диагностики

С твоего ПК (где OCR/embedding):

```powershell
# Что слушает на 8055 и 8001?
netstat -ano | findstr "8055 8001"
# Должно быть LISTENING на 0.0.0.0:8055 и 0.0.0.0:8001

# Если на 127.0.0.1:8055 — значит сервер не слушает LAN, поправь OCR_HOST в start.bat
```

С n8n-сервера или другого ПК офиса:

```powershell
# Достукиваемся до OCR?
curl http://192.168.1.50:8055/status

# Достукиваемся до embedding?
curl http://192.168.1.50:8001/health

# Если виснет — firewall режет. Если ECONNREFUSED — сервер не слушает.
```

С n8n-сервера:

```powershell
# Видит ли n8n твой ПК вообще?
Test-NetConnection -ComputerName 192.168.1.50 -Port 8055
Test-NetConnection -ComputerName 192.168.1.50 -Port 8001
```

`TcpTestSucceeded : True` — порт виден. `False` — firewall или сервер недоступен.

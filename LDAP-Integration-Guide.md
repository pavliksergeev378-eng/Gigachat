# Вход в GigaChat по доменному паролю (через сервис `login_work`) — гайд

Сотрудники входят в GigaChat **тем же логином и паролем, что и в Windows-домене**.
Реализовано **не прямым LDAP**, а через готовый офисный HTTP-сервис аутентификации
`login_work` — он сам проверяет учётку в домене и отвечает «пускать / не пускать».

> **Статус: реализовано и протестировано (E2E).** Осталось одно: после первого
> реального входа в офисе — сверить формат ответа сервиса для извлечения ФИО
> (см. [§7](#7-фио-из-домена-и-его-настройка)).

## 1. Суть в двух словах

- Пароль проверяется **в домене**, а не в нашей БД.
- В сети уже есть **все учётки** — наш сервис их не создаёт, только спрашивает
  «есть такой / пароль верный?».
- При первом доменном входе мы **заводим у себя «тень» пользователя** (чтобы
  привязать сессии/историю), но **доменный пароль у нас не хранится** — вход
  возможен только через домен.
- **Гибрид:** если у пользователя есть локальный пароль (например, у админа,
  заведённого вручную) — сначала проверяется он; это же даёт страховку, если
  офисный сервис недоступен.

Меняется **одна ветка** в воркфлоу авторизации (`sso.json`, действие `login`).
Токены, сессии, все агенты и фронт работают как раньше.

## 2. Как это для пользователя

1. Открывает GigaChat → попадает на `login.html`.
2. Вводит **доменный логин** (формата `efremov_ga`) и **доменный пароль**.
3. «Войти» → внутри. ✅
   - Регистрироваться не нужно.
   - В шапке показывается **ФИО** («Ефремов Григорий Александрович»), а не логин.
   - Сменил пароль в Windows → в GigaChat сразу новый.

## 3. Контракт сервиса `login_work`

Офисный сервис принимает `POST` с телом **`application/x-www-form-urlencoded`**,
одно поле `data` — а в нём JSON-строка:

```
POST http://130.100.92.170/login_work/
Content-Type: application/x-www-form-urlencoded

data={"prefix":"user","action":"create","prop":{"username":"efremov_ga","password":"<пароль>","id":"null"}}
```

> ⚠️ `action:"create"` — название действия сервиса; **ничего не создаёт**, просто
> проверяет наличие учётки и пароль (подтверждено офисом).

**Ответ** — JSON. Ключевое поле:
```json
{ "reload": true }    // пускать
{ "reload": false }   // не пускать (нет учётки / неверный пароль)
```
В ответе также есть **ФИО на русском (три слова)** — точный ключ уточняется в
офисе ([§7](#7-фио-из-домена-и-его-настройка)).

> Транспорт — **HTTP без шифрования** (принятая практика изолированной офисной LAN).

## 4. Поток входа (гибрид local-first)

```
login (username, password)
   │
   ▼
[SQL: login verify]  локальный пароль подошёл? ──да──► user_id известен ──┐
   │ нет                                                                  │
   ▼                                                                      │
[POST login_work]                                                         │
   ├ reload:true  → ФИО извлекается, user_id=null (заведём ниже) ─────────┤
   ├ reload:false → «Неверный логин или пароль»  (стоп)                    │
   └ сервис недоступен → «Сервис аутентификации недоступен» (стоп)         │
                                                                          ▼
                                          [SQL: ensure user] upsert по lower(username)
                                          (создаёт «тень» со СЛУЧАЙНЫМ паролем + ФИО,
                                           либо обновляет ФИО/last_login; без дублей)
                                                                          │
                                                                          ▼
                                          [SQL: login insert session] → токен → ответ
```

- «Сервис недоступен» **не** выдаётся за «неверный пароль» — диагностика честная.
- `ensure user` атомарен (одна `WITH … RETURNING`-транзакция); и создание, и
  обновление возвращают `user_id` корректно (проверено E2E, включая первый вход).

## 5. Что изменено в коде

| Файл | Что |
|---|---|
| `Workflow/sso.json` | ветка `login`: узел **«Login prep»** (local-first + POST в `login_work` + извлечение ФИО), новый узел **«SQL: ensure user»** (upsert доменного юзера), **«SQL: login insert session»** (user_id из ensure user), **«Format login OK»** (+`display_name`), **verify token** (+`display_name` в ответ) |
| `База данных/init-db.sql` | колонка `auth_users.display_name` (ФИО) + `ALTER … ADD COLUMN IF NOT EXISTS` для миграции существующей БД |
| `Agents/_shared.js` | `AUTH_DISPLAYNAME_KEY` + `authGetDisplayName()` + проброс `display_name` в `setAuth`/verify; **изоляция данных (`authUserPrefix`) осталась по `username`**, ФИО — только для показа |
| `login.html` | `setAuth(token, username, display_name)` + показ точной ошибки бэкенда |
| `GigaChat-Platform.html` | в pill шапки — ФИО (`getDisplayName()`), иначе логин |

> Доменный пароль **нигде не хранится**: у «тени» в `password_hash` — случайная
> bcrypt-строка, локальный вход доменным паролем невозможен (только через домен).

## 6. Настройка адреса сервиса

Адрес `login_work` зашит в узле **«Login prep»** (`sso.json`), строка:
```js
const LDAP_URL = 'http://130.100.92.170/login_work/';
```
Чтобы сменить адрес — поправьте здесь и переимпортируйте воркфлоу
(`import-workflows.ps1 -Filter sso`). Боевой адрес уже прописан.

---

## 7. ФИО из домена и его настройка

Точный формат ответа сервиса неизвестен, поэтому ФИО достаётся **эвристикой**:
из ответа берётся первая строка вида **«Три Русских Слова с заглавных»**
(`Фамилия Имя Отчество`) — рекурсивно по всем полям JSON. Это работает для
большинства форматов.

**После первого реального входа в офисе:**
1. Узнай реальный ответ сервиса — два способа описаны в
   [Приложении](#приложение-команды-для-офиса-шпаргалка): прямой запрос из
   PowerShell или поле `_ldap_raw` в логе n8n.
2. Пришли мне этот ответ — если ФИО лежит в конкретном поле (а не как «три слова»),
   заменю эвристику на точное извлечение за минуту.

Если ФИО не распозналось — в шапке покажется логин (`efremov_ga`), на вход это
никак не влияет.

## 8. Развёртывание в офисе

1. **БД** (если ещё не накатана новая схема):
   ```bash
   psql -U postgres -d ai_agent -f "База данных/init-db.sql"
   ```
   `init-db.sql` идемпотентен — на существующей БД лишь добавит `display_name`
   (`ALTER … ADD COLUMN IF NOT EXISTS`), данные не тронет.
2. **Воркфлоу**:
   ```powershell
   .\import-workflows.ps1 -Filter sso
   ```
   Postgres-credential «Postgres» привязывается автоматически.
3. **Фронт** — статические файлы (`login.html`, `Agents/_shared.js`,
   `GigaChat-Platform.html`) разложить как обычно (Ctrl+F5 в браузере).
4. Проверить, что с сервера n8n виден `http://130.100.92.170/login_work/`
   (он в той же LAN).

## 9. Тест-кейсы (как проверить)

Все проверены локально через мок `login_work`; в офисе повторить с реальным сервисом.

| # | Действие | Ожидаемо |
|---|---|---|
| 1 | Доменный логин + верный пароль (юзера ещё нет у нас) | вход; в `auth_users` появилась «тень» с ФИО; в шапке ФИО |
| 2 | Тот же логин ещё раз | вход; **дублей нет** (upsert) |
| 3 | Доменный логин + неверный пароль | «Неверный логин или пароль» |
| 4 | Сервис `login_work` выключен | «Сервис аутентификации недоступен» (НЕ «неверный пароль») |
| 5 | Локальный админ (есть локальный пароль) | вход без обращения к домену |
| 6 | F5 / переход между агентами | ФИО в шапке сохраняется (через verify) |
| 7 | «Выйти» | сессия и ФИО очищены |

Быстрый ручной тест (PowerShell, нужен поднятый n8n):
```powershell
$U='http://localhost:5678/webhook/planner-auth'
Invoke-RestMethod -Uri $U -Method Post -ContentType 'application/json; charset=utf-8' `
  -Body (@{action='login'; username='efremov_ga'; password='<доменный>'} | ConvertTo-Json)
# → { response='ok'; token=...; display_name='Ефремов Григорий Александрович' }  (или error)
```

## 10. Безопасность

- **Доменный пароль у нас не хранится** (даже в хэше) — только проверяется в домене.
- «Тень» юзера нужна лишь для привязки сессий/истории (`user_id`).
- **Изоляция данных** между пользователями — по `username` (не по ФИО), как и раньше.
- Сессионные токены и проверка — без изменений (живая сессия в `auth_sessions`).
- HTTP к `login_work` — по изолированной LAN (по решению офиса).
- Гибрид: локальный пароль даёт админ-доступ, если домен/сервис недоступны.

## 12. FAQ

**В шапке логин, а не ФИО.** Сервис не вернул распознаваемое ФИО — пришли мне его
ответ из execution n8n ([§7](#7-фио-из-домена-и-его-настройка)).

**«Сервис аутентификации недоступен».** `login_work` не отвечает (выключен /
не виден из сети n8n). Проверь доступность `http://130.100.92.170/login_work/`
с сервера n8n. Админ при этом может войти локальным паролем.

**Доменный юзер хочет сменить пароль в GigaChat.** Нельзя — доменный пароль
меняется в Windows. Кнопка «Сменить пароль» работает только для локальных учёток.

**Сменили адрес сервиса.** Поправь `LDAP_URL` в узле «Login prep» и переимпортируй
воркфлоу ([§6](#6-настройка-адреса-сервиса)).

**Нужен прямой LDAP вместо `login_work`.** Не требуется: сервис уже инкапсулирует
домен. Если когда-нибудь понадобится — это отдельная ветка с теми же точками
интеграции (Login prep → ensure user).

## Приложение: команды для офиса (шпаргалка)

### А1. Миграция БД — две колонки

Если не хочется накатывать весь `init-db.sql`, добавьте две колонки в `auth_users`
(команды безопасны, повторный запуск ничего не ломает):

```bash
# 1) ФИО из домена (для LDAP-входа)
psql -U postgres -d ai_agent -c "ALTER TABLE auth_users ADD COLUMN IF NOT EXISTS display_name VARCHAR(200);"

# 2) роль администратора — ОБЯЗАТЕЛЬНА, если деплоится compliance (загрузчик Базы знаний).
#    Без неё verify-токена упадёт и вход сломается у всех!
psql -U postgres -d ai_agent -c "ALTER TABLE auth_users ADD COLUMN IF NOT EXISTS is_admin BOOLEAN DEFAULT false;"
```

> Если Postgres в Docker (как дома): `docker exec postgres psql -U admin -d ai_agent -c "..."`
> — команда внутри кавычек та же.

> ⚠️ **Важно:** обе колонки добавляются и проверяются в `verify`-запросе SSO. Если в
> офисе была накатана только `display_name`, добавьте `is_admin` второй командой —
> иначе после деплоя compliance-изменений вход перестанет работать.

### А2. Как точно узнать, в каком поле сервис отдаёт ФИО

Нужно один раз посмотреть **полный ответ** `login_work` на успешный вход.
Два способа — хватит любого.

**Способ 1 — прямой запрос из PowerShell (без n8n, с любого офисного ПК):**

```powershell
$json = '{"prefix":"user","action":"create","prop":{"username":"СВОЙ_ЛОГИН","password":"СВОЙ_ПАРОЛЬ","id":"null"}}'
$body = 'data=' + [uri]::EscapeDataString($json)
Invoke-RestMethod -Uri 'http://130.100.92.170/login_work/' -Method Post `
  -Body $body -ContentType 'application/x-www-form-urlencoded' | ConvertTo-Json -Depth 10
```

Подставь свой доменный логин/пароль — в консоли появится весь JSON-ответ сервиса.
Где-то в нём будет ФИО («Фамилия Имя Отчество») — запомни **имя поля**, в котором
оно лежит.

**Способ 2 — через лог n8n (после первого реального входа через GigaChat):**

1. Открой n8n → workflow **«[GigaChat] SSO. Поток»** → вкладка **Executions**.
2. Выбери последний успешный запуск → кликни узел **«Login prep»**.
3. В его выводе (Output) есть поле **`_ldap_raw`** — это и есть полный сырой ответ
   сервиса (добавлен специально для этой диагностики; на фронт он не уходит).

**Что дальше с этим знанием:**

- Если ФИО в ответе — обычная строка «Три Русских Слова», то **ничего делать
  не надо**: текущая эвристика его уже поймает (проверь шапку — там должно быть
  ФИО, а не логин).
- Если в шапке остался логин — пришли мне скопированный ответ (пароль из него
  можно затереть), и я заменю эвристику на точное чтение нужного поля в узле
  «Login prep» (`Workflow/sso.json`). Правка одной строки:
  `display_name: findFio(ans)` → `display_name: String(ans.<имя_поля> || '')`,
  затем `import-workflows.ps1 -Filter sso`.

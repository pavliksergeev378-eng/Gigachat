# RAG: подключение внешней базы документов (rel_db)

RAG-агент умеет работать **одновременно с двумя источниками документов**:
- **Наша** `documents` в `ai_agent` — то что загружали через «Загрузка документов» на дашборде (чанки: один файл = много строк)
- **Внешняя** `_vectordocuments.v_general` в `rel_db` — база которая уже наполнена в офисе

Объединение работает в ДВУХ режимах:
1. **Семантический поиск** («найди про X», «что в документах о Y») — эмбеддинг запроса ищет похожие чанки в обеих базах, мержит по `cosine_similarity`, топ-5 уходит в LLM.
2. **Список / подсчёт** («сколько документов», «какие документы», «покажи файлы») — считает документы из обеих баз.

Если внешний источник не настроен (нет credential) — внешние ноды тихо возвращают ошибку (`onError`), RAG продолжает работать **только по нашей** `documents`. Регрессий нет.

---

## Координаты внешней базы (известны, вписаны в workflow)

| Параметр | Значение |
|---|---|
| Database | `rel_db` (host / user / password — **те же** что у `ai_agent`) |
| Схема | `_vectordocuments` |
| Таблица | `v_general` |
| Полное имя | `_vectordocuments.v_general` |
| Размерность вектора | `1024` (та же модель эмбеддингов что у нас) |

**Столбцы `v_general`:** `id`, `vector`, `text`, `document_id`, `file_name`, `chunk_index`, `total_chunks`, `metadata`, `uploaded_at`.

**Маппинг (их → наши):**
| `v_general` | наша `documents` | Зачем |
|---|---|---|
| `file_name` | `filename` | имя файла в ответе |
| `text` | `chunk_text` | содержимое для LLM |
| `vector` | `embedding` | для `cosine_distance` |
| `document_id` | — | группировка документов (подсчёт) |

`document_id`, `total_chunks`, `metadata` в поиске не используются. `document_id` используется для **подсчёта** документов.

> `"text"` и `"vector"` в SQL закавычены — это имена-типы PostgreSQL, в кавычках трактуются как имена столбцов.

## Что нужно сделать в офисе (один раз)

### Шаг 1. Создать credential `Postgres rel_db` в n8n

1. n8n UI → **Credentials** → **New** → **Postgres**
2. host / port / user / password — **те же** что у существующей `Postgres`
3. **database** → `rel_db`
4. Имя credential → **`Postgres rel_db`** (точно так — скрипт ищет по этому имени)
5. Сохранить

### Шаг 2. Привязать credential + обновить workflow

```powershell
git pull
.\import-workflows.ps1 -ResetCreds
```

В выводе должно быть:
```
postgres / Postgres rel_db -> id: <какой-то-uuid>
```
и **НЕ должно быть** блока `⚠️ ВРЕМЕННЫЕ FALLBACK CREDENTIALS`.

Если блок есть — значит credential `Postgres rel_db` не создан или назван иначе. Вернись к Шагу 1.

### Шаг 3. Проверить

В RAG-чате:
- «**сколько документов**» → должно показать число документов из внешней базы (+ наши, если есть)
- «**найди про <тема которая есть только во внешней базе>**» → осмысленный ответ из их документов

Готово.

## Диагностика структуры таблицы (опционально)

Подсчёт документов сделан **универсальным** — группировка по `document_id`, поэтому работает независимо от того, как устроена таблица (один документ = одна строка, ИЛИ документ разбит на чанки). Тебе НЕ нужно это определять вручную.

Но если хочешь понять структуру (например для отладки), прогони в DBeaver к `rel_db`:

```sql
-- Статистика структуры
SELECT
  COUNT(*)                      AS всего_строк,
  COUNT(DISTINCT document_id)   AS уникальных_document_id,
  COUNT(DISTINCT file_name)     AS уникальных_file_name,
  MAX(total_chunks)             AS макс_total_chunks,
  MAX(chunk_index)              AS макс_chunk_index
FROM _vectordocuments.v_general;
```

Как читать:
| Результат | Структура | Что покажет «сколько документов» |
|---|---|---|
| `всего_строк` == `уникальных_document_id` | 1 документ = 1 строка | число строк (= число документов) |
| `всего_строк` > `уникальных_document_id` | документ разбит на чанки | число уникальных `document_id` (= число документов) |

В обоих случаях подсчёт `GROUP BY document_id` даёт **число документов**, а не строк/чанков — это то что нужно.

```sql
-- Посмотреть 5 реальных строк
SELECT document_id, file_name, chunk_index, total_chunks, LEFT(text, 80) AS text_начало
FROM _vectordocuments.v_general
ORDER BY document_id, chunk_index
LIMIT 5;
```

> ⚠️ Если `document_id` бывает `NULL` — все NULL-строки сгруппируются в один «документ». Маловероятно (обычно это NOT NULL / PK), но если так — скажи, поправлю на другой ключ.

## Архитектура (как устроено в workflow)

**Семантический поиск:**
```
Эмбеддинг вопроса (один раз)
       ↓
Построить поиск
   ├→ Поиск в documents (ai_agent)   ─┐
   └→ Поиск внешний (rel_db)          ┴→ Merge внешний+наш → Объединить документы (top-5 по similarity) → Проверка → LLM
```

**Список / подсчёт документов:**
```
Классификатор (видит «сколько/какие/список документов»)
       ↓
Switch [list_documents]
   ├→ SELECT DISTINCT documents (ai_agent, GROUP BY filename) ─┐
   └→ Список внешний (rel_db, GROUP BY document_id)            ┴→ Merge списки → Формат списка документов → Ответ
```

**SQL внешнего поиска** (нод «Поиск внешний»):
```sql
SELECT file_name AS filename, chunk_index, "text" AS chunk_text,
       1 - ("vector" <=> $1::vector) AS similarity, 'внешний' AS source
FROM _vectordocuments.v_general
ORDER BY "vector" <=> $1::vector LIMIT 5
```

**SQL внешнего списка** (нод «Список внешний»):
```sql
SELECT COALESCE(NULLIF(MAX(file_name), ''), 'Документ #' || document_id::text) AS filename,
       COUNT(*) AS chunks, MIN(uploaded_at) AS uploaded_at
FROM _vectordocuments.v_general
GROUP BY document_id
ORDER BY MIN(uploaded_at) DESC NULLS LAST LIMIT 1000
```

Все внешние ноды (`Поиск внешний`, `Список внешний`) имеют `onError: continueRegularOutput` — если credential не настроен или БД упала, они тихо отдают пусто, RAG работает только по нашей `documents`.

## Troubleshooting

**«сколько документов» показывает мало / 0:**
- Открой последний execution в n8n → нод «Список внешний». Если 0 строк или error — credential не привязан или имя таблицы не то.
- Проверь в DBeaver: `SELECT COUNT(DISTINCT document_id) FROM _vectordocuments.v_general;`

**«сколько документов» показало странное число (напр. «5») при пустой `documents`:**
- Это был старый баг (R8.42): запрос «количество документов» не распознавался как подсчёт, уходил в семантический поиск, возвращал топ-5 чанков, LLM говорил «5». Исправлено — «количество/кол-во/число документов» теперь распознаётся как подсчёт.

**Семантический поиск не находит / «не нашёл документов»:**
- Порог similarity ≥ 0.5. Если вектора их базы сделаны ДРУГОЙ моделью эмбеддингов (не той что у нас) — `cosine_distance` вернёт мусор, ничего не пройдёт порог. Проверь что модель та же.
- Проверь execution → «Поиск внешний» → есть ли строки с similarity ≥ 0.5.

**Ошибка `column "vector"/"text" does not exist`:**
- Имя столбца в их таблице другое. Скажи реальное — поправлю SQL.

**`Cannot publish workflow: Credential not configured`:**
- Credential `Postgres rel_db` не существует в n8n. Создай (Шаг 1) и перезапусти `import-workflows.ps1 -ResetCreds`.

## Что НЕ делается (by design)

- **Document-loader не пишет во внешнюю таблицу** — она read-only с нашей стороны. Наша загрузка идёт только в `documents`.
- **Дедупликация между базами не делается** — если файл с одним именем есть в обеих базах, посчитается дважды. Наша `documents` сейчас пустая, так что неактуально.
- **Семантический поиск по целым документам vs чанкам:** если их `text` хранит документ целиком (не чанк), при поиске вернётся весь документ — промпт LLM может раздуться на больших документах. Если станет проблемой (локальный LLM виснет) — скажи, добавлю обрезку `LEFT(text, N)`.

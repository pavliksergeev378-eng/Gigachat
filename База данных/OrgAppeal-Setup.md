## 1. Что мы создаём

В БД `ai_agent` появятся 3 таблицы:

| Таблица | Назначение | Кол-во строк (тест) |
|---------|------------|---------------------|
| `appeal_employees` | Реестр сотрудников: ФИО + табельный номер (nullable!) | 60 |
| `appeal_event1` | Журнал **Мероприятия №1** — для каждого сотрудника выполнено/нет | 40 |
| `appeal_event2` | Журнал **Мероприятия №2** — аналогично | 28 |

Имена таблиц зашиты в `Workflow/organization-appeal.json`. Менять их без правки workflow **нельзя**.

**Главное отличие от v1**: в `appeal_employees` колонка `employee_number` теперь **nullable** — есть сотрудники в реестре, у которых табельный номер ещё не присвоен. Это позволяет различать «сотрудник не в реестре» (D-сценарий) и «сотрудник есть, но без табельного» (E-сценарий).

## 2. Логика алгоритма

```
┌─────────────────────────────────────────────────────────────────┐
│   ФАЗА 1. ИЗВЛЕЧЕНИЕ из документа                                │
│   .docx → JSZip / .pdf → OCR → LLM                              │
│   Извлекаются: ФИО, Дата рождения, Личный номер (ХХХ-ХХХ),       │
│                СНИЛС (ХХХ-ХХХ-ХХХ ХХ)                            │
│                                                                  │
│   ФИО НЕТ            → ❌ «ФИО заявителя отсутствует»            │
│   ДР НЕТ             → ❌ «Дата рождения заявителя отсутствует»  │
│   Нет ЛН И нет СНИЛС → ❌ «Личный номер или СНИЛС отсутствует»   │
│   Иначе → служебная записка идентифицирована, идём дальше       │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│   ФАЗА 2. ИДЕНТИФИКАЦИЯ ЗАЯВИТЕЛЯ (БД №1: appeal_employees)      │
│                                                                  │
│   ФИО НЕ в реестре      → ❌ «Заявитель не идентифицирован»      │
│   ФИО есть, ТН = NULL    → ❌ «Сведений недостаточно»            │
│   ФИО + ТН              → ✅ табельный номер записывается        │
│                            на странице, идём дальше              │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│   ФАЗА 3. МЕРОПРИЯТИЕ №1 (БД №2: appeal_event1)                  │
│                                                                  │
│   ФИО НЕ в таблице         → ❌ «Сведения отсутствуют»           │
│   ФИО есть, is_done=FALSE  → ❌ «Мероприятие не выполнено»       │
│   ФИО есть, is_done=TRUE   → ✅ идём дальше                      │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│   ФАЗА 4. МЕРОПРИЯТИЕ №2 (БД №3: appeal_event2)                  │
│                                                                  │
│   ФИО НЕ в таблице         → ❌ «Сведения отсутствуют»           │
│   ФИО есть, is_done=FALSE  → ⚠️  «Обратиться за помощью»          │
│   ФИО есть, is_done=TRUE   → 🎉 «Все мероприятия пройдены»       │
└─────────────────────────────────────────────────────────────────┘
```

10 финальных исходов алгоритма (A–J) покрываются 10 тест-документами в `Tests/OrgAppeal/`.

## 4. Создание таблиц

Скопируй и выполни блок целиком:

```sql
-- =============================================================
-- ТАБЛИЦА 1: Реестр сотрудников
-- =============================================================
-- employee_number — NULLABLE: сотрудник может быть в реестре,
-- но без присвоенного табельного номера (E-сценарий).
CREATE TABLE IF NOT EXISTS appeal_employees (
    id              SERIAL PRIMARY KEY,
    full_name       VARCHAR(200) NOT NULL,
    employee_number VARCHAR(50),          -- NULLABLE
    created_at      TIMESTAMP DEFAULT NOW()
);

-- Уникальность по ФИО — один человек = одна запись
CREATE UNIQUE INDEX IF NOT EXISTS idx_appeal_employees_name
    ON appeal_employees (full_name);


-- =============================================================
-- ТАБЛИЦА 2: Журнал Мероприятия №1
-- =============================================================
CREATE TABLE IF NOT EXISTS appeal_event1 (
    id          SERIAL PRIMARY KEY,
    full_name   VARCHAR(200) NOT NULL,
    is_done     BOOLEAN NOT NULL DEFAULT FALSE,
    done_at     TIMESTAMP,
    created_at  TIMESTAMP DEFAULT NOW()
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_appeal_event1_name
    ON appeal_event1 (full_name);


-- =============================================================
-- ТАБЛИЦА 3: Журнал Мероприятия №2
-- =============================================================
CREATE TABLE IF NOT EXISTS appeal_event2 (
    id          SERIAL PRIMARY KEY,
    full_name   VARCHAR(200) NOT NULL,
    is_done     BOOLEAN NOT NULL DEFAULT FALSE,
    done_at     TIMESTAMP,
    created_at  TIMESTAMP DEFAULT NOW()
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_appeal_event2_name
    ON appeal_event2 (full_name);


-- Проверим
\dt appeal_*
```

После `\dt appeal_*` должны быть видны все 3 таблицы.

## 6. Проверка готовности

```sql
-- 1. Кол-во строк
SELECT 'appeal_employees' AS table_name, COUNT(*) AS rows FROM appeal_employees
UNION ALL
SELECT 'appeal_event1',     COUNT(*) FROM appeal_event1
UNION ALL
SELECT 'appeal_event2',     COUNT(*) FROM appeal_event2;
-- Ожидаемо: 60 / 40 / 28

-- 2. Сколько сотрудников без табельного номера (для E-сценария)
SELECT COUNT(*) FROM appeal_employees WHERE employee_number IS NULL;
-- Ожидаемо: 4 (Сидоров + 3 балласта)

-- 3. Полный успех (J-сценарий) — кто прошёл всю цепочку
SELECT e.full_name, e.employee_number
FROM appeal_employees e
JOIN appeal_event1 ev1 ON ev1.full_name = e.full_name AND ev1.is_done = TRUE
JOIN appeal_event2 ev2 ON ev2.full_name = e.full_name AND ev2.is_done = TRUE
WHERE e.employee_number IS NOT NULL
ORDER BY e.full_name;
-- Должны быть как минимум: Иванова Иванна Ивановна (наш главный 🎉),
-- а также около 13 других для статистики.
```

## 7. Тестовые сценарии

В репозитории лежат 10 пар (`.docx` + `.pdf`) в [`Tests/OrgAppeal/`](Tests/OrgAppeal/). Каждая пара покрывает одну ветку алгоритма.

| Сцен. | DOCX / PDF | Что в документе | Ожидаемый итог |
|---|---|---|---|
| A | `01-no-fio.docx`              | ❌ ФИО | «ФИО заявителя отсутствует, нужно предоставить данные» |
| B | `02-no-birth.docx`            | ❌ ДР  | «Дата рождения заявителя отсутствует, нужно предоставить данные» |
| C | `03-no-id.docx`               | ❌ ЛН и СНИЛС оба | «Личный номер или СНИЛС отсутствует» |
| D | `04-not-in-db.docx`           | всё есть, ФИО Кузнецов | «Заявитель не идентифицирован» |
| E | `05-no-tab-number.docx`       | всё есть, ФИО Сидоров | «Сведений недостаточно» (ТН=NULL) |
| F | `06-not-in-event1.docx`       | всё есть, ФИО Богданов | Шаг 3: «Сведения отсутствуют» (нет в event1) |
| G | `07-event1-not-done.docx`     | всё есть, ФИО Морозов | Шаг 3: «Мероприятие №1 не выполнено» |
| H | `08-not-in-event2.docx`       | всё есть, ФИО Михайлов | Шаг 4: «Сведения отсутствуют» (нет в event2) |
| I | `09-event2-not-done.docx`     | всё есть, ФИО Васильев | Шаг 4: «Необходимо обратиться за помощью» |
| **J** | `10-full-success.docx`    | всё есть, ФИО Иванова | 🎉 «Все мероприятия пройдены успешно» |

Прогнав все 10 — проверяешь **все** ветки алгоритма.

### Как запустить тест

1. Открой **Дашборд GigaChat → Алгоритмы → Организация обращения**.
2. Перетащи файл из `Tests/OrgAppeal/` в зону загрузки.
3. Нажми «Обработать обращение».
4. Сверь итог с колонкой «Ожидаемый итог».

### Сводка по сотрудникам и обоим мероприятиям

```sql
SELECT
    e.full_name,
    COALESCE(e.employee_number, '— нет —') AS employee_number,
    CASE
        WHEN ev1.id IS NULL  THEN 'нет записи'
        WHEN ev1.is_done     THEN 'выполнено'
        ELSE                      'не выполнено'
    END AS event1_status,
    CASE
        WHEN ev2.id IS NULL  THEN 'нет записи'
        WHEN ev2.is_done     THEN 'выполнено'
        ELSE                      'не выполнено'
    END AS event2_status
FROM appeal_employees e
LEFT JOIN appeal_event1 ev1 ON ev1.full_name = e.full_name
LEFT JOIN appeal_event2 ev2 ON ev2.full_name = e.full_name
ORDER BY e.full_name;
```

# Отчёт о проделанной работе

**Проект:** GigaChat — корпоративная AI-платформа на базе n8n  
**Отдел:** Развития и внедрения перспективных технологий  
**Дата:** 14.06.2026  
**Статус:** Прототип развёрнут, функциональность подтверждена

---

## 1. Введение

В период проведения сборов по вопросам внедрения технологий искусственного интеллекта перед группой поставлена задача развернуть прототип системы автоматизации обработки запросов с использованием платформы n8n, отечественной ИИ-модели GigaChat и базы данных PostgreSQL.

Ожидаемый итог — создание сквозного конвейера: приём сообщения от пользователя через веб-интерфейс, передача в n8n, обработка моделью, сохранение результатов в БД и выдача ответа.

В ходе работы проект эволюционировал от простого чата с LLM до многокомпонентной платформы с набором специализированных агентов.

### 2.2. Компоненты

| Компонент | Технология | Назначение |
|---|---|---|
| **HTTP-сервер** | Caddy (портативный, ~40 МБ) | Раздача статики, порт 8765 |
| **Оркестратор** | n8n | Воркфлоу, интеграции, логика |
| **LLM** | GigaChat API | Генерация ответов, обработка запросов |
| **СУБД** | PostgreSQL + pgvector | Хранение данных, сессий и векторных эмбеддингов (19 таблиц) |
| **Python в браузере** | Pyodide (WebAssembly) | Математические вычисления (NumPy, SciPy, SymPy) |
| **Распознавание речи** | Vosk (русская модель) | Голосовой ввод, полностью офлайн |
| **OCR-сервер** | FastAPI + EasyOCR + PyMuPDF | Извлечение текста из PDF/скан/картинок (порт 8055) |
| **Embedding-сервер** | FastAPI + SentenceTransformers | Векторизация текста (модель e5-large, dim=1024, порт 8001) |
| **Доменная аутентификация** | HTTP-сервис `login_work` | Вход по доменному паролю (гибрид: домен / локальная БД) |
| **Plane-интеграция** | Plane API (Docker) | Управление проектами и задачами через NL-интерфейс |

### 2.3. Ключевые особенности архитектуры

- **Полная офлайн-работоспособность** — ни один компонент не требует доступа в интернет
- **Portable-запуск** — достаточно распаковать ZIP и запустить `GigaChat-Start.bat`
- **Zero-установка** — не требуется прав администратора или установки зависимостей
- **Кроссплатформенность** — Windows (GigaChat-Start.bat) и Linux (GigaChat-Start.sh, systemd-сервис)

### 4.5. Создание экосистемы агентов (текущая фаза)

В ходе развития проекта разработаны следующие агенты:

#### Чат-агент (`chat-agent.html`)
Базовый интерфейс чата с LLM. Поддержка множества сессий, боковая панель с историей, контекстное меню, Markdown-разметка ответов, подсветка кода, автоподбор высоты поля ввода, тёмная и светлая темы.

#### RAG-агент (`rag-agent.html`)
Поиск по документам с использованием Retrieval-Augmented Generation. Позволяет загружать документы и задавать вопросы по их содержимому. Поддержка .txt, .pdf, .docx. Включает векторное индексирование и поиск.

#### Math-агент (`math-agent.html`)
Математические вычисления в браузере через Pyodide (WebAssembly). Поддерживает численные расчёты (NumPy), символьные вычисления (SymPy), научные расчёты (SciPy). Полностью офлайн — Python-рантайм выполняется на стороне клиента.

#### Compliance-агент (`compliance-agent.html`)
Проверка документов на соответствие нормативным требованиям. Загрузка документов, анализ по заданным правилам, формирование отчёта о несоответствиях.

#### OCR-агент (`OCR-search-FIO.html`)
Распознавание текста на изображениях/сканах документов и поиск ФИО. Интеграция с PDF.js для работы с PDF-файлами, выделение текста из отсканированных документов.

#### Агент организации обращений (`organization-appeal.html`)
Обработка организационных запросов. Проверка полноты данных (ФИО, дата рождения, табельный номер), верификация по базам событий, маршрутизация обращений.

#### Text Extractor (`text-extractor.html`)
Извлечение чистого текста из различных форматов файлов (.docx, .pdf, .txt, .csv и др.) с использованием JSZip и PDF.js.

#### Prompt Engineer (`prompt-engineer.html`)
Инструмент для разработки и тестирования промптов. Позволяет быстро проверять формулировки запросов, сравнивать ответы, сохранять шаблоны.

#### Дополнительные агенты
- **Document Loader** — загрузка документов в систему
- **File Merger** — объединение файлов
- **Table Merger** — объединение табличных данных

#### Plane-агент (`plane-agent.html`)
Интеграция с open-source платформой управления проектами Plane (Docker). Позволяет через естественно-языковой чат:
- просматривать проекты и задачи (фильтрация по статусу, приоритету, исполнителю);
- создавать, изменять, удалять задачи;
- управлять приоритетами и сроками (умный парсинг: «срочно», «когда будет время», «до пятницы»);
- выполнять bulk-операции (создать N задач за один запрос);
- работать с эпиками и подзадачами;
- формировать сводки, статистику, дайджесты за день.

Поддерживает многоходовые диалоги (уточнение проекта, местоимения «ей/её», sticky-проект при открытой доске) и контекстное меню.

Разработан подробный тестовый набор — **72 тест-кейса** (OFFICE-TESTING.md) для ручного регрессионного тестирования, покрывающий дымовые тесты, идентичность, проекты, создание/изменение/удаление задач, приоритеты, bulk-операции, многоходовые диалоги, статистику и граничные случаи.

### 4.8. Вспомогательные сервисы

#### OCR-сервер (порт 8055)
Отдельный Python-сервис на FastAPI для извлечения текста из документов. Полностью офлайн, работает в корпоративной LAN.

**Возможности:**
- PDF (с текстовым слоем — быстро через PyMuPDF; сканированные — через EasyOCR с паддингом на русском и английском);
- DOCX (python-docx), картинки (PNG/JPG/TIFF/BMP/WEBP), TXT/CSV/MD;
- Fallback-механизм: если PyMuPDF извлёк менее 50 символов — применяется OCR.

**Развёртывание:**
- Ставится в отдельную папку (`C:\ocr-server\`), вне проекта GigaChat;
- Офлайн-установка через wheels.zip (~260 МБ) из GitHub Releases;
- Модели EasyOCR (~95 МБ) — отдельно, скачиваются один раз;
- Поддерживает автозапуск через папку автозагрузки, Task Scheduler или NSSM (как Windows-служба).

#### Embedding-сервер (порт 8001)
Python-сервис на FastAPI для векторизации текста с использованием модели **intfloat/multilingual-e5-large** (dim=1024).

**API:**
- `POST /embed` — один текст → один вектор (1024 float);
- `POST /embed_batch` — несколько текстов сразу (OpenAI-совместимый формат);
- `GET /health` — проверка живости.

**Использование в проекте:**
- `Workflow/document-loader.json` — векторизация кусков текста при загрузке документов (тип `passage`);
- `Workflow/rag-agent.json` — векторизация пользовательского вопроса при поиске (тип `query`).

**Развёртывание:**
- Отдельная папка `C:\embedding-server\`, модель в `C:\models\multilingual-e5-large\`;
- Офлайн-установка через wheels.zip (~204 МБ) из GitHub Releases;
- Поддержка Linux (systemd) и Windows (Task Scheduler);
- torch CPU (GPU не требуется).

### 4.9. Голосовой ввод (Vosk)

Во все чат-агенты платформы встроена кнопка голосового ввода, использующая **Vosk** — полностью офлайн-систему распознавания речи.

**Как работает:**
- Модель `vosk-model-small-ru.tar.gz` (русский язык) загружается с локального Caddy-сервера;
- Запись микрофона → Vosk (WASM, в браузере) → распознанный текст в поле ввода;
- Вся обработка — на стороне клиента, ни один байт аудио не покидает ПК пользователя.

**Особенность для офисной сети:**
Стандартный браузер блокирует микрофон на небезопасных контекстах (`http://`, кроме `localhost`). Решение — групповая политика `OverrideSecurityRestrictionsOnInsecureOrigin`. Реализованы два способа настройки:

| Способ | Действие | Права |
|---|---|---|
| **.reg-файл** | Двойной клик → «Да» → перезапуск браузера | Не требует админа (HKCU) |
| **PowerShell-скрипт** | `enable-mic-http.ps1` ставит политику на весь ПК | Администратор (HKLM) |
| **PsExec** | `deploy-all-psexec.ps1` — массовая раскатка на список ПК | Администратор сети |

При открытии агента по HTTP платформа автоматически генерирует `.reg`-файл для скачивания — пользователь может включить микрофон в один клик, без вызова IT-специалиста.

### 4.10. Доменная аутентификация (LDAP / login_work)

Реализован вход в GigaChat по доменному паролю — сотрудники используют те же учётные данные, что и для входа в Windows.

**Архитектура:**
- Не прямой LDAP, а через существующий офисный HTTP-сервис `login_work`;
- Сервис проверяет учётную запись в домене и отвечает «пускать / не пускать»;
- Пароль не хранится в БД платформы — только проверяется через домен.

**Гибридная схема:**
1. Сначала проверяется локальный пароль (для администраторов, заведённых вручную);
2. Если локальной учётки нет — запрос к доменному сервису `login_work`;
3. При недоступности доменного сервиса — fallback на локальную аутентификацию.

**Пользовательский опыт:**
- При первом входе в `login.html` пользователь вводит доменный логин и пароль;
- В шапке агента отображается ФИО из домена (например, «Иванов Иван Иванович»), а не логин;
- Смена пароля в Windows — пароль в GigaChat обновляется автоматически (на следующий вход);
- При первом доменном входе создаётся «тень» пользователя в БД для привязки сессий и истории.

**Статус:** E2E-протестировано. Требуется уточнение формата поля ФИО в ответе офисного сервиса.

## 5. Выявленные проблемы и принятые меры

| № | Проблема | Принятое решение | Статус |
|---|---|---|---|
| 1 | На выходе штатного триггера Chat отсутствовала необходимая служебная информация, что нарушало целостность схемы. | Триггер Chat заменён на Webhook, сопряжённый с отдельной HTML-страницей чата. | Решено |
| 2 | Не удавалось установить соединение с локально развёрнутым GigaChat. | Экспериментальным подбором структуры JSON-запроса найдена и устранена причина сбоя, интеграция выполнена успешно. | Решено |
| 3 | Все элементы интерфейса n8n представлены на английском языке. | Рекомендовано проведение работ по русификации интерфейса для повышения удобства операторов. | Предложено руководству |
| 4 | Math-агент не работал при открытии через `file://` — Pyodide требует HTTP-origin для загрузки WASM. | Добавлен портативный Caddy-сервер с корректными MIME-типами для `.wasm` и `.zip`. | Решено |
| 5 | После обновления файлов в офисе браузер показывал старую версию из кэша. | Настроены заголовки Cache-Control: `no-store` для HTML и JS (кроме библиотек). | Решено |
| 6 | В офисной сети Caddy не отдавал корректный Content-Type для WASM-файлов. | Добавлено явное правило `header Content-Type application/wasm` для `.wasm-запросов`. | Решено |
| 7 | Утечка данных через клиентский кэш при логине. | Реализован механизм `gc-pending-auth`: контент скрыт до верификации токена. | Решено |
| 8 | **Микрофон не работает на HTTP-сайтах.** Браузер блокирует доступ к микрофону на `http://<ip>` (требуется secure context — HTTPS или localhost). | Разработаны скрипты и .reg-файлы для установки политики `OverrideSecurityRestrictionsOnInsecureOrigin` на ПК пользователя (см. раздел 4.9). | Решено |
| 9 | **EasyOCR недоступен при старте сервера.** Модели EasyOCR грузятся лениво — первый запрос к OCR-серверу может длиться 10–30 секунд. | Это штатное поведение. В README даны пояснения; рекомендовано «прогревать» сервер тестовым запросом после запуска. | Принято как особенность |
| 10 | **PDF без текстового слоя не распознавался.** Сканированные документы (PDF как картинка) не давали полезного текста. | Разработан OCR-сервер с EasyOCR: PyMuPDF извлекает текстовый слой (если есть), при недостаточном количестве символов (<50) — fallback на EasyOCR. | Решено |
| 11 | **Embedding-модель требует интернета для загрузки.** Первый запуск SentenceTransformers пытается скачать модель из HuggingFace Hub. | Создан механизм офлайн-бандлов: wheels.zip (~204 МБ) и модель с ПК с интернетом переносятся флешкой в LAN. HuggingFace offline-флаги (`HF_HUB_OFFLINE=1`) заданы в server.py. | Решено |
| 12 | **Монолитная структура общего кода.** Файл `_shared.js` (325 КБ, ~5500 строк) содержит весь общий код для всех агентов в одном файле. Каждый HTML-агент подключает его целиком, хотя использует лишь часть функционала. | Предложены варианты решений (см. раздел 5.1). | Требует оценки |

### 5.1. Подробный разбор проблемы №8: монолитная структура `_shared.js`

Файл **`Agents/_shared.js`** занимает **325 КБ (5467 строк кода)** и выполняет роль «швейцарского ножа» — в нём сосредоточена вся общая логика платформы: авторизация, HTTP-клиент, Markdown-рендеринг, чат-интерфейс, работа с документами, загрузка файлов, управление темами, обработчики событий и т.д. Каждый HTML-агент подключает его через `<script src="_shared.js">` и получает **весь файл целиком**, независимо от того, какой функционал ему нужен.

#### 5.1.1. Актуальна ли проблема для корпоративной сети?

Оценка проведена применительно к условиям офисной LAN-инфраструктуры:

| Фактор | Оценка |
|---|---|
| **Скорость передачи в LAN** | 1 Гбит/с → 325 КБ передаётся за ~3 мс. На практике с учётом MTU, TCP-накладных и HTTP-заголовков — 10–30 мс. |
| **Парсинг JS в браузере** | На современном ПК (Core i5 + SSD): ~50–100 мс. На офисных ПК (Pentium/Celeron, HDD, Chrome 90-х версий): ~200–500 мс. |
| **Суммарная задержка** | На слабых машинах может достигать **0,5–1 секунды** на каждый запуск агента. |
| **Накопительный эффект** | Если оператор в течение дня 50 раз открывает разных агентов — теряется ~25–50 секунд рабочего времени только на парсинг `_shared.js`. |
| **Расход ОЗУ** | После парсинга весь код остаётся в памяти браузера, даже если агент использует только 20–30% функций. Для офисных ПК с 4–8 ГБ ОЗУ и десятком открытых вкладок — дополнительная нагрузка. |

**Вердикт:** для локальной сети проблема не критична, но на слабых офисных рабочих станциях или при частом переключении между агентами даёт ощутимый накопленный эффект. Масштабирование решения (например, добавление новых агентов) будет увеличивать `_shared.js` дальше, усугубляя ситуацию.

#### 5.1.2. Варианты решения

Три подхода с разным соотношением «затраты / результат»:

| Вариант | Описание | Плюсы | Минусы |
|---|---|---|---|
| **А. Модульная структура** | Разделить `_shared.js` на тематические файлы: `_auth.js`, `_chat-ui.js`, `_http-client.js`, `_markdown.js`, `_file-utils.js` и т.д. Каждый агент подключает только то, что использует. | Прозрачная архитектура; экономия загрузки 40–80% объёма на каждый агент; легко добавлять новые модули | Требуется рефакторинг ~1–2 дня; нужно проверить все 11 агентов на корректность зависимостей |
| **Б. Lazy-loading (динамическая загрузка)** | `_shared.js` превращается в загрузчик (~20 КБ), который подтягивает модули по требованию: `GigaChat.loadModule('chat-ui')` — модуль загружается в момент первого вызова его функций. | Минимальная загрузка при старте; модули подгружаются только когда реально нужны; не требует правки HTML-файлов | Сложнее отладка (асинхронная загрузка); первый вызов любой функции имеет задержку на подгрузку модуля |
| **В. Оставить как есть** | Ничего не менять. `_shared.js` продолжит загружаться целиком на каждой странице. | Ноль затрат времени; текущий код стабилен | С ростом проекта файл будет только расти; проблема не решится сама собой |

#### 5.1.3. Рекомендация

Для текущего этапа проекта (прототип, ограниченное число пользователей) — **Вариант В** (оставить как есть) является оправданным. Влияние на производительность в локальной сети минимально, и затраты времени на рефакторинг на данном этапе нецелесообразны.

Рекомендуется вернуться к вопросу при выполнении одного из условий:
- количество агентов превысит 15;
- появятся жалобы пользователей на скорость загрузки;
- планируется тиражирование платформы на более широкий круг пользователей.

В этом случае наиболее сбалансированным представляется **Вариант А** (модульная структура) — он даёт прозрачную архитектуру и не вносит асинхронных усложнений при отладке.
